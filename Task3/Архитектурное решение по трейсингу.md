# Архитектурное решение по трейсингу

## 1. Анализ системы: точки, где заказ может «сломаться» или зависнуть

### Жизненный цикл заказа и критические переходы

Заказ проходит через три системы: **Онлайн-магазин (Shop API)**, **CRM (CRM API)** и **MES (MES API)**, взаимодействующие через **RabbitMQ** и **Shop DB** (общая БД для Shop API и CRM API).

```
Customer/API user
       │
       ▼
  ┌──────────┐     ┌──────────┐     ┌──────────┐
  │ Shop API │────▶│ Shop DB  │◀────│ CRM API  │
  └──────────┘     └──────────┘     └────┬─────┘
       │                                 │
       │           ┌──────────┐          │
       ▼           │ RabbitMQ │◀─────────┘
  ┌──────────┐     │          │─────────▶┌──────────┐
  │    S3    │◀────│          │          │ MES API  │
  └──────────┘     └──────────┘          └────┬─────┘
                                              │
                                         ┌────▼─────┐
                                         │  MES DB  │
                                         └──────────┘
```

### Точки отказа (идентифицированы по C4-диаграмме)

| # | Переход | Статус заказа | Тип риска | Описание |
|---|---------|---------------|-----------|----------|
| 1 | Customer → Shop API | `INITIATED` → `FILE_UPLOADED` | Потеря данных | Загрузка 3D-файла: Shop API сохраняет файл в S3 и запись в Shop DB. Сбой между этими двумя операциями — файл загружен, но заказ не обновлён (или наоборот). |
| 2 | Shop API → S3 | `FILE_UPLOADED` | Таймаут/ошибка | S3 недоступен или таймаут при загрузке тяжёлого 3D-файла. Клиент не знает, дошёл ли файл. |
| 3 | Shop API → Shop DB | `SUBMITTED` | Ошибка записи | Клиент нажал «Сделать заказ», но запись в БД не прошла (таймаут, deadlock, исчерпание соединений — CRM API конкурирует за ту же БД). |
| 4 | CRM API → RabbitMQ | `SUBMITTED` → `PRICE_CALCULATED` | **Потеря сообщения** | CRM API отправляет заказ на расчёт стоимости в MES API через RabbitMQ. Сообщение может быть потеряно при отсутствии publisher confirms. **Самая частая точка потери заказов.** |
| 5 | RabbitMQ → MES API | `SUBMITTED` → `PRICE_CALCULATED` | **Потеря при обработке** | MES API получил сообщение, но consumer упал до завершения обработки. Без правильного ack — сообщение потеряно. |
| 6 | MES API: расчёт стоимости | `PRICE_CALCULATED` | Зависание | Расчёт стоимости занимает 2-30 минут. Таймаут, OOM или падение инстанса во время расчёта — заказ застревает навсегда. |
| 7 | MES API → RabbitMQ | `PRICE_CALCULATED` → CRM | **Потеря сообщения** | MES API отправляет результат расчёта обратно в CRM API через RabbitMQ. Аналогичный риск потери. |
| 8 | RabbitMQ → CRM API | Обновление статуса | **Потеря при обработке** | CRM API получает обновление статуса, но не может записать в Shop DB (deadlock, таймаут). |
| 9 | API user → MES API | Весь цикл | Потеря запроса | B2B-партнёр отправляет заказ напрямую в MES API (без gateway). При перегрузке MES API запрос может быть отклонён без уведомления. |
| 10 | MES API → MES DB | Все статусы MES | Ошибка записи | Обновление статуса в MES DB не прошло. Заказ в одном состоянии в MES DB, в другом — в RabbitMQ/Shop DB. |

### Системы, которые необходимо покрыть трейсингом

На C4-диаграмме выделены **красным** следующие контейнеры:

- **Shop API** — точка входа для B2C-заказов, загрузка файлов.
- **CRM API** — маршрутизация заказов, подтверждение, связь с RabbitMQ.
- **MES API** — точка входа для B2B-заказов, расчёт стоимости, обновление статусов.
- **RabbitMQ** — центральный узел обмена сообщениями между CRM API и MES API.
- **Shop DB** — общая БД для Shop API и CRM API (важно трейсить запросы от обоих сервисов).
- **MES DB** — БД производства.
- **S3** — хранение 3D-файлов.

### Данные, которые должны попадать в трейсинг

Для каждого span в трейсе должны фиксироваться:

**Обязательные атрибуты:**
- `trace_id` — сквозной идентификатор, единый для всего жизненного цикла заказа.
- `span_id` — идентификатор конкретной операции.
- `parent_span_id` — связь с родительской операцией.
- `service.name` — имя сервиса (`shop-api`, `crm-api`, `mes-api`).
- `operation` — имя операции (например, `create_order`, `upload_file`, `calculate_price`, `publish_message`, `consume_message`).
- `timestamp` — время начала и окончания операции.
- `duration` — длительность операции.
- `status` — результат (`OK`, `ERROR`).
- `error.message` — текст ошибки (при наличии).

**Бизнес-атрибуты:**
- `order.id` — идентификатор заказа.
- `order.status` — текущий статус заказа (`INITIATED`, `FILE_UPLOADED`, `SUBMITTED` и т.д.).
- `order.source` — источник заказа (`b2c`, `b2b`).
- `customer.id` — идентификатор клиента (или `partner.id` для B2B).

**Атрибуты для RabbitMQ-сообщений:**
- `messaging.system` — `rabbitmq`.
- `messaging.destination` — имя exchange/очереди.
- `messaging.message_id` — идентификатор сообщения.
- `messaging.operation` — `publish` или `process`.

**Атрибуты для БД-операций:**
- `db.system` — `postgresql`.
- `db.name` — имя базы данных.
- `db.operation` — тип операции (`SELECT`, `INSERT`, `UPDATE`).
- `db.statement` — SQL-запрос (с маскированием чувствительных данных).

## 2. Мотивация

### Почему нужен трейсинг

Главная бизнес-проблема «Александрита» — потеря заказов. Клиенты B2C и B2B-партнёры не получают свои заказы, а команда не может определить, где именно в цепочке обработки заказ «застрял» или потерялся. Разбор инцидентов ведётся со слов клиента и занимает часы или дни.

Заказ проходит через три независимых сервиса (Shop API, CRM API, MES API) и асинхронную очередь (RabbitMQ). Без сквозного трейсинга невозможно:

1. Определить, в каком сервисе находится заказ прямо сейчас.
2. Понять, на каком переходе заказ потерялся.
3. Измерить время обработки на каждом этапе.
4. Отличить проблему в коде от проблемы в инфраструктуре (БД, RabbitMQ, S3).

### Метрики, на которые повлияет внедрение трейсинга

**Технические метрики:**

1. **MTTR (Mean Time to Resolve)** — среднее время устранения инцидента. Сейчас: часы/дни (разбор по жалобам). Ожидаемое: минуты (поиск заказа по `trace_id` → мгновенная локализация проблемы).
2. **MTTD (Mean Time to Detect)** — среднее время обнаружения проблемы. На основе данных трейсинга можно настроить алерт на заказы, «зависшие» в статусе дольше порога.
3. **Error rate по переходам между статусами** — доля заказов, которые не смогли перейти в следующий статус. Сейчас эта метрика вообще не измеряется.

**Бизнес-метрики:**

4. **Процент потерянных заказов** — доля заказов, не дошедших до статуса `CLOSED`. Трейсинг позволит точно измерить и отслеживать динамику.
5. **Время выполнения заказа (order lead time)** — полное время от `INITIATED` до `CLOSED`. Трейсинг покажет, где «бутылочные горлышки», и позволит целенаправленно оптимизировать.

## 3. Предлагаемое решение

### Выбор технологии

**OpenTelemetry (OTel)** как стандарт инструментирования + **Jaeger** как бэкенд для хранения и визуализации трейсов.

| Критерий | Решение | Обоснование |
|---|---|---|
| Стандарт инструментирования | **OpenTelemetry SDK** | Вендор-нейтральный стандарт (CNCF). Есть библиотеки для Java (Shop API, CRM API) и .NET (MES API). Поддерживает автоматическое инструментирование HTTP, gRPC, SQL, RabbitMQ. |
| Бэкенд трейсинга | **Jaeger** | Open-source, нативная поддержка OpenTelemetry, горизонтально масштабируется. Хорошая визуализация трейсов с Flame Graph и DAG. |
| Коллектор | **OpenTelemetry Collector** | Центральный агент для приёма, обработки и экспорта телеметрии. Позволяет менять бэкенд без изменения кода приложений. Может семплировать трейсы для снижения нагрузки. |
| Хранилище трейсов | **Elasticsearch** (или OpenSearch) | Jaeger поддерживает Elasticsearch как storage backend. Позволяет хранить трейсы с полнотекстовым поиском по атрибутам (order_id, customer_id). |

### Новые компоненты

1. **OpenTelemetry Collector** — отдельный инстанс (EC2), принимает телеметрию от всех сервисов по протоколу OTLP, обрабатывает и отправляет в Jaeger.
2. **Jaeger** (query + collector + storage) — инстанс для хранения и визуализации трейсов.
3. **Elasticsearch** — storage backend для Jaeger (может использоваться совместно с системой логирования из Задания 4).

### Инструментирование сервисов

**Shop API и CRM API (Java Spring Boot):**
- Подключить `opentelemetry-javaagent` — Java-агент, который автоматически инструментирует Spring MVC (HTTP-запросы), JDBC (SQL-запросы), RabbitMQ client (publish/consume).
- Добавить кастомные span-ы для бизнес-операций: создание заказа, загрузка файла, смена статуса.
- Добавить бизнес-атрибуты (`order.id`, `order.status`, `order.source`) в span-ы.

**MES API (C#/.NET):**
- Подключить `OpenTelemetry.Extensions.Hosting` и `OpenTelemetry.Instrumentation.AspNetCore` — автоматическое инструментирование ASP.NET Core.
- Добавить `OpenTelemetry.Instrumentation.SqlClient` для трейсинга SQL-запросов к MES DB.
- Подключить инструментирование RabbitMQ-клиента.
- Добавить кастомные span-ы для расчёта стоимости (начало, завершение, ошибка) с атрибутами длительности и сложности модели.

**Пропагация контекста через RabbitMQ:**
- При публикации сообщения в RabbitMQ: инжектировать `trace_id` и `span_id` в headers сообщения (стандарт W3C Trace Context).
- При получении сообщения: извлечь `trace_id` из headers и продолжить трейс. Это обеспечит **сквозной трейс через асинхронную очередь**.

### Схема данных трейса (пример для B2C-заказа)

```
Trace: order-12345
│
├── [Shop API] POST /api/orders (create_order)
│   ├── [Shop API] INSERT INTO orders (db.query)
│   └── [Shop API] Upload file to S3 (s3.upload)
│
├── [Shop API] POST /api/orders/12345/submit (submit_order)
│   └── [Shop API] UPDATE orders SET status='SUBMITTED' (db.query)
│
├── [CRM API] Read order from Shop DB (poll/event)
│   └── [CRM API] Publish to RabbitMQ: calculate_price (messaging.publish)
│
├── [MES API] Consume from RabbitMQ: calculate_price (messaging.process)
│   ├── [MES API] Download file from S3 (s3.download)
│   ├── [MES API] Calculate price (business.calculate_price) — 2-30 min
│   ├── [MES API] UPDATE orders SET status='PRICE_CALCULATED' (db.query → MES DB)
│   └── [MES API] Publish to RabbitMQ: price_calculated (messaging.publish)
│
├── [CRM API] Consume from RabbitMQ: price_calculated (messaging.process)
│   └── [CRM API] UPDATE orders SET status='PRICE_CALCULATED' (db.query → Shop DB)
│
├── [CRM API] Approve manufacturing (manual action by Seller)
│   ├── [CRM API] UPDATE orders SET status='MANUFACTURING_APPROVED' (db.query → Shop DB)
│   └── [CRM API] Publish to RabbitMQ: manufacturing_approved (messaging.publish)
│
├── [MES API] Consume: manufacturing_approved (messaging.process)
│   ... (MANUFACTURING_STARTED → COMPLETED → PACKAGING → SHIPPED)
│
└── [CRM API] Close order
    └── [CRM API] UPDATE orders SET status='CLOSED' (db.query → Shop DB)
```

Ссылка на обновлённую диаграмму: `Task3/jewerly_c4_model_tracing.drawio`

Новые компоненты (выделены **красным**):

- **OpenTelemetry Collector** — принимает телеметрию от Shop API, CRM API, MES API.
- **Jaeger** — бэкенд трейсинга, UI для визуализации.
- **Elasticsearch** — хранилище трейсов.
- Новые связи (красные стрелки):
  - Shop API → OTel Collector (OTLP)
  - CRM API → OTel Collector (OTLP)
  - MES API → OTel Collector (OTLP)
  - OTel Collector → Jaeger (OTLP)
  - Jaeger → Elasticsearch (storage)
  - Support/DevTeam → Jaeger UI (просмотр трейсов)

## 4. Компромиссы

### 4.1. Расчёт стоимости (2-30 минут)

Длительные операции расчёта создают span-ы, которые могут жить до 30 минут. Это не проблема для хранения, но:
- В UI Jaeger такие трейсы выглядят неинформативно — один огромный span доминирует над остальными.
- **Решение:** разбить расчёт на подэтапы (загрузка файла, парсинг модели, расчёт, сохранение результата) с отдельными span-ами внутри.

### 4.2. Ручные операции

Некоторые переходы статусов зависят от действий людей:
- Продавец подтверждает заказ в CRM (`PRICE_CALCULATED` → `MANUFACTURING_APPROVED`).
- Оператор берёт заказ в работу в MES (`MANUFACTURING_APPROVED` → `MANUFACTURING_STARTED`).

Время между этими действиями может составлять часы или дни. Трейсинг здесь **не покажет техническую проблему** — задержка обусловлена бизнес-процессом. Но трейс зафиксирует момент перехода, что полезно для анализа.

### 4.3. MES — купленный софт

MES был куплен вместе с исходным кодом. Инструментирование MES API потребует:
- Модификации кодовой базы на C# (подключение OTel SDK).
- Один разработчик C# в команде (+ тимлид — бывший C#-разработчик).
- **Риск:** изменения в чужом коде могут быть трудоёмкими, если архитектура MES не предусматривает middleware/interceptor-ов.
- **Компромисс:** на первом этапе ограничиться автоматическим инструментированием (ASP.NET Core, SQL, HTTP-клиент). Кастомные бизнес-span-ы добавлять поэтапно.

### 4.4. Overhead на производительность

Трейсинг добавляет overhead:
- Каждый span — это сериализация + отправка по сети.
- Для нагруженных эндпоинтов overhead может составлять 1-5% latency.
- **Компромисс:** использовать семплирование на OTel Collector. Для начала — 100% трейсов (нагрузка пока умеренная). При росте — переход на adaptive sampling (всегда трейсить ошибки, семплировать успешные).

### 4.5. S3 — ограниченная наблюдаемость

S3 (Yandex Object Storage) — managed-сервис. Нельзя установить агент внутри. Трейсинг ограничен:
- Можно трейсить вызовы **к** S3 из Shop API и MES API (клиентская сторона).
- Нельзя трейсить, что происходит **внутри** S3.
- **Компромисс:** достаточно клиентских span-ов — они покажут latency и ошибки при работе с хранилищем.

## 5. Аспекты безопасности

### 5.1. Контроль доступа к Jaeger UI

- **Аутентификация:** доступ к Jaeger UI только для сотрудников компании с актуальной учётной записью. Интеграция с корпоративным SSO/LDAP (если есть) или базовая аутентификация (OAuth2 Proxy перед Jaeger UI).
- **Авторизация:** доступ ограничен ролями:
  - **Поддержка / DevOps** — полный доступ к просмотру трейсов (поиск по order_id, customer_id).
  - **Разработчики** — доступ к трейсам в dev и release окружениях, ограниченный доступ к prod.
  - **Продавцы и операторы** — нет доступа к системе трейсинга.
- **Сетевая изоляция:** Jaeger UI доступен только из внутренней сети компании (VPN/VPC). Не должен быть доступен из интернета.

### 5.2. Защита чувствительных данных в трейсах

- **Маскирование PII:** персональные данные клиентов (имя, email, телефон, адрес) не должны попадать в атрибуты span-ов. Использовать только идентификаторы (`customer_id`, `order_id`).
- **Маскирование SQL-запросов:** при записи `db.statement` заменять значения параметров на плейсхолдеры (`SELECT * FROM orders WHERE customer_id = ?`), чтобы не хранить данные клиентов в трейсах.
- **Фильтрация на OTel Collector:** настроить процессор `attributes` для удаления/маскирования чувствительных атрибутов до отправки в Jaeger. Это централизованная точка контроля.
- **Не логировать тела запросов/ответов** — они могут содержать персональные и финансовые данные.

### 5.3. Безопасность транспорта

- **OTLP over gRPC с TLS:** трафик от сервисов к OTel Collector шифруется.
- **Внутренняя сеть:** OTel Collector и Jaeger развёрнуты в том же VPC, что и приложения. Трейсы не покидают периметр облака.

### 5.4. Хранение и ротация

- **Retention policy:** трейсы хранятся 14 дней (достаточно для расследования инцидентов). Elasticsearch автоматически удаляет индексы старше 14 дней.
- **Шифрование at rest:** Elasticsearch использует шифрование дисков (managed-сервис Yandex Cloud).

## 6. Автоматический мониторинг прохождения заказа и алертинг (дополнительное задание)

### Концепция

На основе данных трейсинга можно автоматически отслеживать «здоровье» процесса прохождения заказа. Идея: из трейсов извлекаются бизнес-метрики, на которые настраиваются алерты.

### Реализация

**Компонент: SpanMetrics Connector в OTel Collector**

OTel Collector поддерживает `spanmetrics` connector, который автоматически генерирует RED-метрики (Rate, Errors, Duration) из span-ов и экспортирует их в Prometheus. Это позволяет получить метрики без дополнительного кода в приложениях.

**Метрики, извлекаемые из трейсов:**

1. **Количество заказов, зависших в статусе дольше порога:**
   - `SUBMITTED` → `PRICE_CALCULATED`: порог 40 минут (максимальное время расчёта 30 мин + запас).
   - `PRICE_CALCULATED` → `MANUFACTURING_APPROVED`: порог 24 часа (ручная операция продавца).
   - `MANUFACTURING_APPROVED` → `MANUFACTURING_STARTED`: порог 48 часов.

2. **Error rate по переходам:** доля span-ов со статусом ERROR на каждом переходе.

3. **Доля заказов в DLX:** количество сообщений в dead-letter-exchange / общее количество отправленных сообщений.

**Алертинг (через Prometheus Alertmanager → Grafana):**

| Алерт | Условие | Severity | Действие |
|-------|---------|----------|----------|
| Заказ завис в `SUBMITTED` | Нет span `PRICE_CALCULATED` > 40 мин | Critical | Тикет + уведомление дежурного. Проверить MES API и RabbitMQ. |
| Рост error rate на RabbitMQ publish | Error rate > 1% за 5 мин | Critical | Тикет + звонок. Проверить доступность RabbitMQ. |
| Рост latency расчёта стоимости | p99 > 35 мин | Warning | Тикет. Проверить нагрузку MES API, количество заказов в очереди. |
| Заказы попадают в DLX | DLX count > 0 | Critical | Тикет + звонок. Каждое сообщение в DLX = потенциально потерянный заказ. |
| Нет трейсов от сервиса | 0 span-ов от сервиса > 5 мин | Critical | Тикет + звонок. Сервис, вероятно, упал. |

### Обновлённая диаграмма

Новые компоненты (выделены **зелёным**):

- **Prometheus** — получает метрики из SpanMetrics Connector OTel Collector.
- **Alertmanager** — обрабатывает алерты из Prometheus.
- **Grafana** — визуализация метрик, полученных из трейсов, и управление алертами.
- Новые связи (зелёные стрелки):
  - OTel Collector → Prometheus (spanmetrics)
  - Prometheus → Alertmanager (алерты)
  - Alertmanager → Notification channels (Telegram/Slack/PagerDuty)
  - Grafana → Prometheus (визуализация)
  - Grafana → Jaeger (drill-down из метрик в конкретные трейсы)
