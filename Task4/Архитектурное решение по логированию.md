# Архитектурное решение по логированию

## 1. Анализ системы: какие логи нужно собирать

### Системы, из которых требуется сбор логов (по C4-диаграмме)

На C4-диаграмме отмечены все контейнеры, из которых необходим сбор логов:

- **Shop API** (Spring Boot) — точка входа B2C-клиентов, загрузка 3D-файлов, создание заказов.
- **CRM API** (Spring Boot) — маршрутизация заказов, подтверждение, обмен сообщениями через RabbitMQ. Использует **Shop DB** (общую БД с Shop API).
- **MES API** (C#/.NET) — точка входа B2B-партнёров, расчёт стоимости, управление производством.
- **RabbitMQ** — обмен сообщениями между CRM API и MES API.
- **Shop DB** (PostgreSQL) — общая БД для Shop API и CRM API.
- **MES DB** (PostgreSQL) — БД производства.

### Логи уровня INFO (бизнес-операции)

#### Shop API

| Событие | Данные для логирования |
|---------|----------------------|
| Создание заказа | `timestamp`, `order_id`, `customer_id`, `source: b2c`, `status: INITIATED` |
| Загрузка 3D-файла | `timestamp`, `order_id`, `customer_id`, `file_name`, `file_size_kb`, `storage_path` (S3 key), `status: FILE_UPLOADED` |
| Создание 3D-модели в конструкторе | `timestamp`, `order_id`, `customer_id`, `polygon_count`, `status: FILE_UPLOADED` |
| Отправка заказа | `timestamp`, `order_id`, `customer_id`, `status: SUBMITTED` |
| Обновление статуса заказа (получено от CRM/MES) | `timestamp`, `order_id`, `old_status`, `new_status` |

#### CRM API

| Событие | Данные для логирования |
|---------|----------------------|
| Получение нового заказа (из Shop DB) | `timestamp`, `order_id`, `customer_id`, `seller_id` (если назначен) |
| Публикация сообщения в RabbitMQ | `timestamp`, `order_id`, `queue_name`, `message_id`, `action` (например, `request_price_calculation`) |
| Получение сообщения из RabbitMQ | `timestamp`, `order_id`, `queue_name`, `message_id`, `action` (например, `price_calculated`) |
| Подтверждение производства | `timestamp`, `order_id`, `seller_id`, `status: MANUFACTURING_APPROVED` |
| Закрытие заказа | `timestamp`, `order_id`, `seller_id`, `status: CLOSED`, `close_reason` (`delivery_confirmed` / `manual`) |
| Обновление статуса заказа в Shop DB | `timestamp`, `order_id`, `old_status`, `new_status` |

#### MES API

| Событие | Данные для логирования |
|---------|----------------------|
| Получение заказа через API (B2B) | `timestamp`, `order_id`, `partner_id`, `source: b2b` |
| Получение сообщения из RabbitMQ | `timestamp`, `order_id`, `queue_name`, `message_id`, `action` |
| Начало расчёта стоимости | `timestamp`, `order_id`, `file_path` (S3 key), `polygon_count` (если доступен) |
| Завершение расчёта стоимости | `timestamp`, `order_id`, `calculated_price`, `calculation_duration_sec`, `status: PRICE_CALCULATED` |
| Публикация сообщения в RabbitMQ | `timestamp`, `order_id`, `queue_name`, `message_id`, `action` |
| Назначение заказа на оператора | `timestamp`, `order_id`, `operator_id`, `status: MANUFACTURING_STARTED` |
| Завершение производства | `timestamp`, `order_id`, `operator_id`, `status: MANUFACTURING_COMPLETED` |
| Начало упаковки | `timestamp`, `order_id`, `operator_id`, `status: PACKAGING` |
| Отправка заказа | `timestamp`, `order_id`, `status: SHIPPED`, `tracking_number` |

#### RabbitMQ

| Событие | Данные для логирования |
|---------|----------------------|
| Сообщение попало в dead-letter-exchange | `timestamp`, `queue_name`, `message_id`, `reason` (`rejected`, `expired`, `maxlen`), `original_routing_key` |
| Consumer подключился/отключился | `timestamp`, `queue_name`, `consumer_tag`, `event` (`connected`/`disconnected`) |

### Другие уровни логирования

#### WARN

- **Повторная обработка сообщения** (retry) — `timestamp`, `order_id`, `queue_name`, `retry_attempt`, `max_retries`, `reason`.
- **Медленный SQL-запрос** (> 1 сек) — `timestamp`, `service`, `query_duration_ms`, `query` (с маскированием параметров).
- **Высокое время расчёта стоимости** (> 15 мин) — `timestamp`, `order_id`, `calculation_duration_sec`.
- **Приближение к лимиту соединений БД** (> 70% от `max_connections`) — `timestamp`, `db_name`, `active_connections`, `max_connections`.
- **Rate limit для B2B-партнёра приближается к порогу** — `timestamp`, `partner_id`, `current_rps`, `limit_rps`.

#### ERROR

- **Ошибка публикации сообщения в RabbitMQ** — `timestamp`, `order_id`, `queue_name`, `error_message`, `stack_trace`.
- **Ошибка обработки сообщения из RabbitMQ** — `timestamp`, `order_id`, `queue_name`, `message_id`, `error_message`, `stack_trace`.
- **Ошибка записи в БД** — `timestamp`, `service`, `operation`, `order_id`, `error_message`, `stack_trace`.
- **Ошибка загрузки/скачивания файла из S3** — `timestamp`, `order_id`, `file_path`, `error_message`.
- **Ошибка расчёта стоимости** — `timestamp`, `order_id`, `error_message`, `stack_trace`.
- **HTTP 5xx ответ** — `timestamp`, `service`, `method`, `endpoint`, `status_code`, `request_id`, `error_message`.
- **Таймаут при обращении к внешнему сервису** — `timestamp`, `service`, `target_service`, `timeout_ms`, `order_id`.

#### DEBUG (только dev и release окружения, не prod)

- Тела HTTP-запросов/ответов (с маскированием PII).
- Полные SQL-запросы с параметрами.
- Содержимое RabbitMQ-сообщений.

## 2. Мотивация

### Почему нужно логирование

Сейчас ошибки и нестандартные ситуации разбираются со слов клиента. Разработчикам и поддержке приходится вручную воспроизводить проблему, подключаться к серверам, искать информацию в разрозненных stdout-логах каждого сервиса. При трёх независимых системах (Shop API, CRM API, MES API) и асинхронном обмене через RabbitMQ этот процесс занимает часы и дни.

Централизованное логирование даст:

- **Единую точку поиска** — вместо подключения к 3+ серверам — один интерфейс для поиска по `order_id`, `customer_id`, временному диапазону.
- **Корреляцию событий** — связывание логов разных сервисов по `order_id` и `trace_id` для восстановления полной картины инцидента.
- **Историю для ретроспективного анализа** — возможность расследовать проблемы, которые произошли дни или недели назад.

### Метрики, на которые повлияет внедрение логирования

**Технические:**

1. **Среднее время расследования инцидента (MTTI)** — с часов/дней до минут. Поиск по `order_id` в централизованной системе вместо ручного разбора по серверам.
2. **Количество неразрешённых инцидентов** — сократится, так как логи дадут информацию для диагностики проблем, которые ранее оставались без объяснения.
3. **Время обнаружения аномалий** — автоматический алертинг на основе анализа логов позволит обнаруживать проблемы до жалоб клиентов.

**Бизнес:**

4. **Удовлетворённость B2B-партнёров** — возможность быстро отвечать на запросы «что с моим заказом» вместо «мы разбираемся, подождите».
5. **Нагрузка на поддержку (время на обработку обращения)** — сократится за счёт быстрого доступа к информации о заказе.

### Приоритет внедрения логирования и трейсинга по системам

Команда не может внедрить логирование и трейсинг во всех системах одновременно. Порядок приоритизации:

| Приоритет | Система | Обоснование |
|-----------|---------|-------------|
| **1** | **MES API** | Центральный узел проблем: расчёт стоимости (зависания), обмен с RabbitMQ (потеря сообщений), точка входа B2B (жалобы партнёров). Максимальная концентрация проблем. |
| **2** | **CRM API + RabbitMQ** | Второй участник обмена сообщениями. CRM API использует Shop DB (общая БД) — важно видеть его запросы. RabbitMQ — место потери сообщений. |
| **3** | **Shop API** | Точка входа B2C. Менее критична, так как основные проблемы (потеря заказов, медленный дашборд) проявляются на стороне MES/CRM. Но важна для полноты картины. |
| **4** | **Shop DB, MES DB** | PostgreSQL-логи (slow queries, ошибки). Добавляются после настройки логирования приложений. |

## 3. Предлагаемое решение

### Выбор технологии

**OpenSearch** (форк Elasticsearch с Apache License 2.0) + **Fluent Bit** (легковесный агент сбора логов) + **OpenSearch Dashboards** (визуализация).

Обоснование выбора — в разделе 5 (дополнительное задание).

### Архитектура решения

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│ Shop API │     │ CRM API  │     │ MES API  │
│ (stdout) │     │ (stdout) │     │ (stdout) │
└────┬─────┘     └────┬─────┘     └────┬─────┘
     │                │                │
     ▼                ▼                ▼
┌──────────┐     ┌──────────┐     ┌──────────┐
│Fluent Bit│     │Fluent Bit│     │Fluent Bit│
└────┬─────┘     └────┬─────┘     └────┬─────┘
     │                │                │
     └────────────────┼────────────────┘
                      ▼
              ┌──────────────┐
              │  OpenSearch   │
              │  (хранилище)  │
              └──────┬───────┘
                     │
              ┌──────▼───────┐
              │  OpenSearch   │
              │  Dashboards   │
              └──────────────┘
```

### Компоненты

1. **Fluent Bit** — устанавливается как sidecar/агент на каждый EC2-инстанс (Shop API, CRM API, MES API). Читает stdout/stderr контейнеров, парсит JSON-логи, добавляет метаданные (hostname, service name) и отправляет в OpenSearch.
2. **OpenSearch** — централизованное хранилище логов. Один кластер (минимум 2 узла: master + data) на отдельных EC2-инстансах.
3. **OpenSearch Dashboards** — UI для поиска и визуализации логов.

### Формат логов

Все сервисы должны писать логи в **структурированном JSON-формате**:

```json
{
  "timestamp": "2025-03-15T14:30:00.123Z",
  "level": "INFO",
  "service": "mes-api",
  "trace_id": "abc123def456",
  "span_id": "789ghi",
  "order_id": "ORD-12345",
  "message": "Price calculation completed",
  "attributes": {
    "calculated_price": 15000,
    "calculation_duration_sec": 180,
    "status": "PRICE_CALCULATED"
  }
}
```

Ключевые поля:
- `trace_id` — связь с трейсингом (из OpenTelemetry). Позволяет перейти из лога в Jaeger для просмотра полного трейса.
- `order_id` — связь с бизнес-процессом. Поиск всех событий по заказу.
- `service` — идентификация источника.

**Для Spring Boot (Shop API, CRM API):** настроить `logback` с JSON-encoder (`logstash-logback-encoder`). MDC (Mapped Diagnostic Context) для `trace_id` и `order_id`.

**Для .NET (MES API):** настроить `Serilog` с JSON-форматированием и обогащением через OpenTelemetry context.

### Инструментирование сервисов

| Сервис | Изменения |
|--------|-----------|
| Shop API (Spring Boot) | Подключить `logstash-logback-encoder`. Добавить MDC-фильтр для `trace_id`, `order_id`. Добавить логирование бизнес-событий (таблица из раздела 1). |
| CRM API (Spring Boot) | Аналогично Shop API. Дополнительно: логирование publish/consume RabbitMQ-сообщений с `message_id`. |
| MES API (C#/.NET) | Подключить `Serilog.Formatting.Compact`. Добавить middleware для обогащения логов `trace_id`, `order_id`. Логирование бизнес-событий. |
| PostgreSQL (Shop DB, MES DB) | Включить `log_min_duration_statement = 1000` (slow queries > 1 сек). Настроить `pgAudit` для аудита DDL-операций. Экспорт через Fluent Bit с PostgreSQL input plugin. |
| RabbitMQ | Включить встроенное логирование. Экспорт через Fluent Bit с файлового input. |

### Политика безопасности

**Чувствительные данные:**

- **Запрещено логировать:** пароли, токены, номера банковских карт, полные email-адреса, телефоны, адреса доставки, содержимое 3D-файлов.
- **Маскирование:** если PII необходимы для отладки (например, email) — маскировать (`i***@example.com`). Реализовать на уровне логгера (custom converter в logback / enricher в Serilog).
- **Идентификаторы вместо данных:** использовать `customer_id`, `order_id`, `partner_id` вместо имён и контактов. При необходимости восстановить данные — запрос в БД по ID.

**Контроль доступа:**

| Роль | Доступ | Обоснование |
|------|--------|-------------|
| Разработчики | Полный доступ к логам всех сервисов в dev/release. Доступ к prod-логам уровня INFO и WARN. | Для отладки и расследования инцидентов. |
| DevOps/SRE | Полный доступ ко всем логам во всех окружениях. | Для управления инфраструктурой. |
| Поддержка | Доступ к prod-логам уровня INFO. Только поиск по `order_id`, `customer_id`. | Для ответа на запросы клиентов без доступа к техническим деталям. |
| Продавцы, операторы | Нет доступа. | Логи содержат внутреннюю техническую информацию. |

**Реализация:** OpenSearch Security Plugin (встроен в OpenSearch) — RBAC с ролями и tenant-ами. Интеграция с корпоративным SSO.

### Политика хранения

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| Отдельный индекс под каждую систему | Да: `logs-shop-api-YYYY.MM.DD`, `logs-crm-api-YYYY.MM.DD`, `logs-mes-api-YYYY.MM.DD`, `logs-rabbitmq-YYYY.MM.DD`, `logs-db-YYYY.MM.DD` | Раздельное управление retention, доступом, и шардированием. Разный объём логов у разных сервисов. |
| Retention (горячие данные) | **14 дней** | Достаточно для оперативного расследования инцидентов. Основной объём поиска — последние 1-3 дня. |
| Retention (холодные данные) | **90 дней** (сжатые, без реплик) | Для ретроспективного анализа и аудита. Перевод индексов старше 14 дней в cold storage через ISM (Index State Management). |
| Оценка объёма | ~5-10 ГБ/день | При текущей нагрузке (сотни заказов в день), ~1000-5000 лог-записей на заказ (все сервисы), средний размер записи ~1 КБ в JSON. С ростом нагрузки — линейный рост. |
| Ротация индексов | Ежедневная | Один индекс = один день. Упрощает удаление старых данных и управление шардами. |
| Количество шардов | 1 primary + 1 replica (на индекс) | При текущем объёме достаточно. При росте > 50 ГБ/день — увеличить шарды. |

**ISM Policy (автоматическое управление жизненным циклом):**

1. Дни 0-14: hot (полная индексация, реплика, быстрый доступ).
2. Дни 14-90: cold (force merge, удаление реплик, сжатие).
3. День 90+: delete.

### Отражение на C4-диаграмме

Новые компоненты:
- **Fluent Bit** — агенты на инстансах Shop API, CRM API, MES API.
- **OpenSearch** — кластер хранения логов.
- **OpenSearch Dashboards** — UI для поиска и визуализации.

Новые связи:
- Shop API → Fluent Bit → OpenSearch
- CRM API → Fluent Bit → OpenSearch
- MES API → Fluent Bit → OpenSearch
- RabbitMQ → Fluent Bit → OpenSearch
- Shop DB → Fluent Bit → OpenSearch
- MES DB → Fluent Bit → OpenSearch
- Support/DevTeam → OpenSearch Dashboards

## 4. Система анализа логов: алертинг и обнаружение аномалий

### Алертинг

На основе логов необходимо настроить следующие алерты (через OpenSearch Alerting Plugin):

| Алерт | Условие (запрос в OpenSearch) | Severity | Действие |
|-------|-------------------------------|----------|----------|
| Всплеск ошибок RabbitMQ | Количество ERROR-логов с `message` содержащим `rabbitmq` > 10 за 5 мин | Critical | Уведомление дежурного (Telegram/Slack). Проверить доступность RabbitMQ и consumer-ов. |
| Ошибки записи в БД | Количество ERROR-логов с `db.operation` > 5 за 5 мин | Critical | Уведомление дежурного. Проверить состояние Shop DB / MES DB. |
| HTTP 500 всплеск | Количество логов с `status_code: 500` > 20 за 5 мин на одном сервисе | Critical | Уведомление дежурного. Возможная причина: сбой после деплоя, перегрузка. |
| Заказ без смены статуса | Лог `status: SUBMITTED` для `order_id` без последующего лога `status: PRICE_CALCULATED` в течение 40 мин | Warning | Создание тикета. Расследование: MES API получил сообщение? |
| Медленные SQL-запросы | Количество WARN-логов `slow_query` > 20 за 10 мин | Warning | Создание тикета. Анализ запросов, оптимизация индексов. |

### Обнаружение аномалий

**Зачем:** стандартные пороговые алерты не улавливают плавную деградацию или необычные паттерны. Примеры аномалий:

1. **Резкий рост количества заказов** — было 4 записи о создании заказов в секунду, стало 10 000. Возможные причины: DDoS, массовый retry B2B-партнёра, бот.
2. **Изменение паттерна ошибок** — обычно 2-3 ошибки в час, стало 50. Порог алерта ещё не достигнут, но тренд аномальный.
3. **Исчезновение логов от сервиса** — сервис перестал писать логи (упал, зависли).

**Реализация:**

- **OpenSearch Anomaly Detection Plugin** — встроенный ML-инструмент, который строит baseline поведения метрик и обнаруживает отклонения.
- Настроить детекторы на:
  - Количество лог-записей по сервису (обнаружит и всплески, и пропадания).
  - Количество ERROR-записей по сервису.
  - Количество записей о создании заказов (обнаружит DDoS).
- При обнаружении аномалии: алерт в канал дежурного с ссылкой на дашборд OpenSearch Dashboards.

## 5. Выбор технологии (дополнительное задание)

### Критерии выбора

| # | Критерий | Описание |
|---|----------|----------|
| 1 | Лицензия | Стоимость владения, возможность модификации, риски смены лицензии. |
| 2 | Совместимость со стеком | Поддержка Java (Spring Boot), C#/.NET, интеграция с OpenTelemetry. |
| 3 | Порог входа и сообщество | Простота развёртывания и настройки, наличие документации, размер сообщества. |
| 4 | Масштабируемость | Способность справляться с ростом объёма логов при линейном росте нагрузки (+100 заказов/месяц). |
| 5 | Встроенная аналитика | Наличие инструментов для алертинга, обнаружения аномалий, визуализации без дополнительных компонентов. |
| 6 | Стоимость инфраструктуры | Требования к RAM, CPU, диску. Количество компонентов для развёртывания. |
| 7 | Интеграция с трейсингом | Возможность использовать единый бэкенд для логов и трейсов (Jaeger из Задания 3). |

### Сравнительный анализ

| Критерий | ELK (Elastic Stack) | OpenSearch | Splunk | Loki (Grafana) |
|----------|---------------------|------------|--------|----------------|
| **Лицензия** | Elastic License 2.0 (не open-source, ограничения на SaaS). Риск ужесточения условий. | Apache License 2.0 (полностью открытая, нет ограничений). | Проприетарная. Высокая стоимость лицензии. | AGPLv3 (open-source, но copyleft). |
| **Совместимость** | Отличная. Logback, Serilog, Filebeat — зрелые интеграции. OpenTelemetry → Elasticsearch. | Отличная. Полностью совместим с Elasticsearch API. ### Выбор: OpenSearch

**Обоснование:**

1. **Лицензия** — Apache 2.0 без ограничений. В отличие от ELK, нет рисков смены лицензии.
2. **Встроенные бесплатные инструменты** — Alerting Plugin и Anomaly Detection включены в базовую поставку. В ELK аналогичные функции требуют платной подписки.
3. **Совместимость с Jaeger** — OpenSearch может служить единым storage backend и для логов, и для трейсов (Задание 3). Это упрощает инфраструктуру и снижает стоимость.
4. **Совместимость со стеком** — работает с тем же API, что и Elasticsearch. Все интеграции (logback, Serilog, Fluent Bit) работают без изменений.
5. **Стоимость** — при команде из 8 инженеров + 1 DevOps бюджет на лицензии Splunk нецелесообразен. Loki дешевле, но проигрывает в аналитике и полнотекстовом поиске — для расследования инцидентов с заказами критично искать по произвольным полям.
