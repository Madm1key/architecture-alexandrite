# Архитектурное решение по кешированию

## 1. Мотивация

### Проблемы, которые должно решить кеширование

Введение кэширование решит проблему долгой загрузки дашборда MES для операторов.

### Какие элементы системы включить в кеширование

| Элемент                                  | Что кешировать                                                      | Обоснование                                                                                                                             |
|------------------------------------------|---------------------------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------|
| Дашборд MES (список заказов по статусам) | Результат запроса списка заказов с фильтром по статусу и пагинацией | Основная жалоба операторов. Горячие данные, частые повторяющиеся запросы от нескольких операторов с одинаковыми параметрами фильтрации. |
| Детали заказа в MES                      | Карточка заказа (статус, оператор, стоимость, метаданные)           | Операторы часто открывают один и тот же заказ. Данные меняются редко (только при смене статуса).                                        |

---

## 2. Предлагаемое решение

Клиентское кеширование не решает основную проблему - снижение нагрузки на MES DB. Для дашборда MES
критична консистентность между операторами. HTTP-кеш в браузере этого не обеспечивает. Поэтому
оптимальным решением будет использование server-side кеша.

### Выбор паттерна кеширования

#### Анализ паттернов

| Паттерн       | Подходит ли для MES-дашборда                                                                                                                                                                                      |
|---------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Cache-Aside   | Полный контроль над логикой кеширования. Промах кеша не критичен - просто запрос в БД.                                                                                                                            |
| Write-Through | Не подходит. MES DB обновляется из разных источников.                                                                                                                                                             |
| Refresh-Ahead | Избыточен. Требует фонового процесса обновления. Дашборд MES отображает данные по запросу оператора - нет необходимости поддерживать кеш «горячим» для всех возможных комбинаций фильтров. Усложнение без выгоды. |

#### Почему Cache-Aside

1. **Простота реализации.** MES API (C#/.NET) проверяет Redis перед запросом к MES DB. Это
   минимальное изменение в существующем коде - добавление слоя кеширования в сервис чтения заказов.
2. **Контроль инвалидации.** При обновлении статуса заказа (через MES API или RabbitMQ-consumer) -
   явно удаляем затронутые ключи кеша. Это проще, чем инструментировать все точки записи для
   Write-Through.
3. **Устойчивость к отказу Redis.** При недоступности Redis система продолжает работать - просто все
   запросы идут напрямую в MES DB (как сейчас). Для Write-Through падение кеша означает потерю
   данных или необходимость дополнительной синхронизации.
4. **Эффективность при частичном кешировании.** Кешируются только запрошенные комбинации фильтров.
   Не нужно заранее прогревать кеш для всех возможных параметров.

### Ключи кеша

```
mes:orders:status:{status}:page:{page}:size:{size}
```

Примеры:

- `mes:orders:status:MANUFACTURING_STARTED:page:1:size:20` - первая страница заказов в работе.
- `mes:orders:status:MANUFACTURING_APPROVED:page:1:size:20` - первая страница заказов, ожидающих
  начала работы.
- `mes:order:{order_id}` - детали конкретного заказа.

## 4. Стратегия инвалидации кеша

### Анализ стратегий

| Критерий                      | Временная (TTL)                                                                                                                                       | По событию (программная)                                                                       | По ключу (ручная)                                             |
|-------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------|------------------------------------------------------------------------------------------------|---------------------------------------------------------------|
| **Описание**                  | Кеш автоматически истекает через заданное время                                                                                                       | Кеш инвалидируется программно при изменении данных (смена статуса заказа)                      | Администратор или скрипт вручную удаляет ключи кеша           |
| **Актуальность данных**       | Данные могут устареть на время TTL. При TTL=30 сек оператор может не видеть новый заказ до 30 секунд.                                                 | Данные актуальны сразу после изменения. Задержка - миллисекунды.                               | Данные актуальны только после ручного вмешательства.          |
| **Сложность реализации**      | Минимальная. Достаточно указать TTL при SET.                                                                                                          | Средняя. Нужно инструментировать все точки изменения данных (MES API: UI + RabbitMQ consumer). | Минимальная для реализации, но высокая операционная нагрузка. |
| **Нагрузка на БД**            | Равномерная. После истечения TTL - запрос в БД. При большом количестве одновременных запросов и одновременном истечении TTL - риск "thundering herd". | Минимальная. Запрос в БД только при cache miss после инвалидации.                              | Непредсказуемая.                                              |
| **Подходит для дашборда MES** | Частично. Операторам важно видеть **самые новые** заказы для получения вознаграждения. Задержка в 30 сек может быть критичной.                        | **Да.** Мгновенное обновление кеша при смене статуса. Операторы видят актуальные данные.       | Нет. Не масштабируется.                                       |

### Выбранная стратегия: TTL + программная инвалидация (гибридный подход)

Используем обе стратегии одновременно:

1. **Программная инвалидация** - основной механизм. При любом изменении статуса заказа в MES API (из
   UI оператора или из RabbitMQ-сообщения) удаляются:
    - Ключи списков по старому статусу: `mes:orders:status:{old_status}:*`
    - Ключи списков по новому статусу: `mes:orders:status:{new_status}:*`
    - Ключ деталей заказа: `mes:order:{order_id}`

2. **TTL = 30 секунд** - страховочный механизм. Защищает от ситуации, когда программная инвалидация
   не сработала (баг, race condition, обновление из другого источника). Через 30 секунд кеш
   обновится в любом случае.

**Почему гибридный подход:**

- Только TTL: операторы могут не видеть новый заказ до 30 секунд - неприемлемо, вознаграждение
  зависит от скорости.
- Только программная инвалидация: если в коде не покрыт какой-то путь обновления данных - кеш
  никогда не обновится. TTL выступает «предохранителем».
- Гибридный: программная инвалидация обеспечивает мгновенную актуальность, TTL - гарантию на случай
  сбоя.

### Защита от thundering herd

При одновременном истечении TTL для популярного ключа несколько операторов одновременно отправят
запрос в БД. Для защиты:

- **Mutex/lock на ключ:** при cache miss первый запрос ставит lock (через `SET NX` в Redis),
  остальные ждут. Первый запрос загружает данные в кеш, остальные получают из кеша.
- Альтернатива: **stale-while-revalidate** - при истечении TTL отдавать устаревшие данные, пока один
  запрос обновляет кеш в фоне.

## 5. Альтернативные решения (дополнительное задание)

|               | Решение 1: Redis (Cache-Aside)                                                                                                                                                                       | Решение 2: Materialized View в PostgreSQL                                                                                                                                                                                                                   |
|---------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Описание**  | Внешний Redis-кеш перед MES DB. MES API проверяет Redis, при промахе - запрос в БД + запись в кеш. Инвалидация при смене статуса + TTL.                                                              | Создание Materialized View в MES DB для дашборда. Периодический `REFRESH MATERIALIZED VIEW CONCURRENTLY`. Без внешних компонентов.                                                                                                                          |
| **Диаграмма** | MES frontend → MES API → Redis → (miss) → MES DB                                                                                                                                                     | MES frontend → MES API → MES DB (Materialized View)                                                                                                                                                                                                         |
| **Плюсы**     | Кеш в RAM - запросы за миллисекунды. Разгружает MES DB полностью (при cache hit БД вообще не затрагивается). Масштабируется горизонтально (Redis Cluster). Мгновенная инвалидация при смене статуса. | Не требует дополнительной инфраструктуры. Простая реализация (один SQL). Нет проблемы консистентности кеша - данные всегда из БД. Нет сетевого хопа до Redis.                                                                                               |
| **Минусы**    | Дополнительный компонент (Redis) - нужно развернуть, мониторить, обеспечить отказоустойчивость. Необходимость инвалидации (доработка кода). Потребление RAM.                                         | Не снижает нагрузку на инстанс БД - refresh выполняется на том же сервере. При частом refresh (каждые 10 сек) - дополнительная нагрузка на БД. Невозможна мгновенная инвалидация - данные устаревают на интервал refresh. Не масштабируется отдельно от БД. |

### Заключение

**Выбор: Решение 1 (Redis, Cache-Aside).**

Materialized View не решает корневую проблему - MES DB остаётся единой точкой нагрузки. Refresh
выполняется на том же инстансе PostgreSQL, который и так перегружен. Кроме того, Materialized View
не обеспечивает мгновенную инвалидацию - операторы будут видеть устаревшие данные на интервал
refresh, что критично для их вознаграждения.

Redis полностью разгружает MES DB при cache hit (запросы не доходят до БД), масштабируется
независимо, обеспечивает мгновенную инвалидацию при смене статуса заказа. Дополнительная сложность (
развёртывание Redis, доработка кода) оправдана эффектом: latency дашборда сокращается с секунд до
миллисекунд, нагрузка на MES DB снижается кратно.
