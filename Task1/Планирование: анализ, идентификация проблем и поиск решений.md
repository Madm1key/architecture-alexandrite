# Планирование: анализ, идентификация проблем и поиск решений

## 1. Идентификация существующих и потенциальных проблем

### 1.1. Потеря и зависание заказов (критическая)

**Симптомы:** клиенты не получают заказы

**Причины:**

- Отсутствует механизм гарантированной доставки и обработки сообщений;
- Отсутствует трейсинг, непонятно на каком из этапов заказы теряются;
- Отсутствует логирование бизнес операций. О проблемах узнают только по жалобам клиентов.

### 1.2. Общая БД для Shop API и CRM API (критическая)

**Симптомы:** потенциальная деградация производительности и риск каскадных сбоев

**Причины:**

- CRM API и Shop API конкурируют за ресурсы одной БД;
- При сбое БД недоступны оба сервиса.

### 1.3. Один инстанс на каждый сервис и БД (критическая)

**Симптомы:** при падении любого инстанса вся соответствующая подсистема недоступна

**Причины:**

- Каждое приложение имеет по одному инстансу;
- БД работают в режиме single-instance (один инстанс на чтение и запись);
- При падении RabbitMQ полностью останавливается обработка заказов.

### 1.4. Отсутствие управления нагрузкой API (высокая)

**Симптомы:** из-за введения API компанию "завалило заказами"

**Причины:**

- Отсутствие rate-лимитов;
- Не используется backpressure;
- Не используется circuit breaker.

### 1.5. Медленная загрузка дашборда MES (высокая)

**Симптомы:** медленная загрузка дашборда MES со списком заказов

**Причины:**

- Отсутствует кэширование;
- Один инстанс БД, который работает и на запись, и на чтение;
- MES API загружена CPU-bounded операцией расчета стоимости.

### 1.6. Длительный процесс релиза (средняя)

**Симптомы:** постоянно увеличиваются задержки релизов

**Причины:**

- Ручное E2E тестирование;
- Ручной деплой в release и prod окружение.

## 2. Инициативы по устранению проблем

Для приоритизации используется подход MoSCow.

- **Внедрение системы наблюдаемости** (мониторинг + логирование + трейсинг)
    - **Цель:** получение информации о состоянии системы; проактивное реагирование на проблемы.
    - **Приоритет:** Must Have

- **Горизонтальное масштабирование и отказоустойчивость**
    - **Цель:** устранить единую точку отказа, обеспечить возможность роста.
    - **Приоритет:** Must Have

- **Кеширование дашборда MES**
    - **Цель:** решить проблему медленной загрузки страницы операторов.
    - **Приоритет:** Should Have

- **Внедрение API Gateway**
    - **Цель:** защитить систему от перегрузки со стороны B2B-партнёров и каскадных отказов.
    - **Приоритет:** Should Have

- **Разделение БД Shop API и CRM API**
    - **Цель:** устранить связанность, обеспечить независимость сервисов, повысить
      производительность.
    - **Приоритет:** Should Have

- **Автоматизация тестирования и CI/CD**
    - **Цель:** сократить время релизного цикла и повысить качество выпускаемого софта.
    - **Приоритет:** Could Have

- **Оптимизация процесса расчёта стоимости в MES**
    - **Цель:** сократить время ожидания расчёта для клиентов.
    - **Приоритет:** Could Have

### Целевая архитектура через полгода

- Настроена наблюдаемость: Prometheus + Grafana для метрик, OpenSearch для логов,
  OpenTelemetry + Jaeger для трейсинга. Настроенный алертинг.
- Добавлены read-реплики для Shop DB и MES DB.
- Настроен кластер для RabbitMQ.
- Добавлен redis-кеш для дашборда MES.
- Настроен API Gateway перед MES API с rate limiting и квотами.
- Используется backpressure / circuit breaker на межсервисных вызовах.

### Если бы можно было выполнить только три инициативы

1. Внедрение системы наблюдаемости
    - Проактивное обнаружение проблем вместо реактивного;
    - Сокращение времени расследования инцидентов.
2. Горизонтальное масштабирование и отказоустойчивость
    - Устраняет единую точку отказа.
    - Даёт запас по производительности для растущей нагрузки.
    - Read-реплика Shop DB позволяет разделить нагрузку от Shop API и CRM API.
    - Кластер RabbitMQ повышает надёжность доставки сообщений.
    - Read-реплика MES DB частично решает проблему медленного дашборда MES.
3. Кеширование дашборда MES
    - Быстрое решение проблемы медленной загрузки страницы операторов.
    - Снижает нагрузку на MES DB, что может улучшить производительность для всех клиентов.
