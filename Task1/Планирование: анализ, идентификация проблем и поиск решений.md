# Планирование: анализ, идентификация проблем и поиск решений

## Описание текущей архитектуры (по C4-диаграмме)

Система «Jewerly Store System» состоит из следующих контейнеров:

| Контейнер                | Технология                | Назначение                                                           |
|--------------------------|---------------------------|----------------------------------------------------------------------|
| Internet Shop (frontend) | Vue, TypeScript, Three.js | Создание заказа, загрузка/создание 3D-модели                         |
| Shop API (backend)       | Spring Boot               | Список товаров, приём 3D-файлов, хранение информации о заказах       |
| CRM (frontend)           | Vue, TypeScript           | Интерфейс для продавцов                                              |
| CRM API (backend)        | Spring Boot               | Работа с заказами со стороны продавцов                               |
| MES (frontend)           | React, TypeScript         | Интерфейс для операторов: список заказов, назначение                 |
| MES API (backend)        | C#                        | Назначение заказов на операторов, расчёт стоимости                   |
| Shop DB                  | PostgreSQL                | Данные о клиентах и заказах                                          |
| MES DB                   | PostgreSQL                | Данные о заказах, назначениях и статусах с точки зрения производства |
| Messages Queue           | RabbitMQ                  | Обмен сообщениями о заказах и статусах между CRM API и MES API       |
| 3D files storage         | S3                        | Хранилище 3D-файлов для расчёта стоимости                            |

**Ключевые связи (по диаграмме):**

- Customer → Internet Shop → Shop API → Shop DB
- Shop API → S3 (загрузка 3D-файлов)
- Seller → CRM → CRM API → **Shop DB** (CRM использует общую БД магазина, собственной БД у CRM нет)
- CRM API ↔ RabbitMQ (двусторонний обмен)
- API user → MES API (прямое подключение, без gateway)
- Operator → MES → MES API → MES DB
- MES API → S3 (чтение 3D-файлов для расчёта)
- MES API ↔ RabbitMQ (двусторонний обмен)

**Инфраструктура:** все сервисы на EC2 в Yandex Cloud, каждое приложение — один инстанс, БД —
Managed Services по одному инстансу (чтение + запись).

## 1. Идентификация существующих и потенциальных проблем

### 1.1. Потеря и зависание заказов (критическая)

**Симптомы:** клиенты B2C и B2B-партнёры (пользователи API) не получают заказы; заказы находятся в
непонятном состоянии месяцами.

**Корневые причины:**

- Взаимодействие между MES API и CRM API реализовано через RabbitMQ, но отсутствует механизм
  гарантированной доставки и обработки сообщений. Сообщения теряются или не обрабатываются — нет
  мониторинга dead-letter-exchange очередей, нет retry-политик, нет алертов на зависшие сообщения.
- Нет сквозного трейсинга заказа: невозможно определить, на каком этапе и в какой системе заказ
  «застрял». Жизненный цикл заказа проходит по цепочке Shop API → (Shop DB) → CRM API → RabbitMQ →
  MES API → RabbitMQ → CRM API. При сбое на любом из переходов заказ теряется без уведомления.
- Shop API не подключён к RabbitMQ напрямую — он не получает обратную связь о статусе заказа от
  MES/CRM. Клиент онлайн-магазина не информируется о прогрессе заказа после отправки (`SUBMITTED`).
- Нет логирования на уровне бизнес-операций: разбор инцидентов ведётся со слов клиента, что крайне
  неэффективно.

### 1.2. Общая БД для Shop API и CRM API — связанность и конкуренция (критическая)

**Симптомы:** потенциальная деградация производительности и риск каскадных сбоев.

**Корневые причины (по C4-диаграмме):**

- CRM API подключён к Shop DB — собственной БД у CRM нет. Это означает:
    - Shop API и CRM API конкурируют за ресурсы одной БД (соединения, CPU, I/O).
    - Тяжёлый запрос со стороны CRM (например, выборка заказов для подтверждения) может
      заблокировать или замедлить работу онлайн-магазина.
    - Миграции схемы данных требуют координации между двумя командами — изменение для CRM может
      сломать Shop API.
    - Один инстанс БД без реплик делает это узкое место ещё критичнее.

### 1.3. Единая точка отказа — по одному инстансу на каждый сервис и БД (критическая)

**Симптомы:** при падении любого инстанса вся соответствующая подсистема недоступна.

**Корневые причины:**

- Каждое приложение имеет по одному инстансу EC2. При его падении или деплое сервис полностью
  недоступен.
- БД работают в режиме single-instance (один инстанс на чтение и запись). Нет репликации, нет
  read-реплик. При сбое Shop DB — недоступны и онлайн-магазин, и CRM одновременно (общая БД).
- RabbitMQ — один инстанс. При его падении полностью останавливается обмен сообщениями между CRM API
  и MES API, заказы перестают обрабатываться.
- Отсутствует балансировка нагрузки — нет возможности горизонтального масштабирования.

### 1.4. Отсутствие API Gateway и защиты публичного API (высокая)

**Симптомы:** после открытия API «компанию завалило заказами», жалоб стало ещё больше.

**Корневые причины (по C4-диаграмме):**

- API user подключается напрямую к MES API без промежуточного слоя. Нет API Gateway, нет rate
  limiting, нет квот.
- MES API одновременно обслуживает и внутренних операторов (через MES frontend), и внешних
  B2B-партнёров. Перегрузка от B2B напрямую влияет на работу операторов.
- Нет backpressure — при перегрузке система принимает все запросы, но не справляется с их
  обработкой.
- Нет circuit breaker между сервисами — сбой MES API каскадно влияет на RabbitMQ и далее на CRM API.

### 1.5. Медленная загрузка дашборда MES (высокая)

**Симптомы:** операторы жалуются, что первая страница MES с фильтрами и пагинацией долго
прогружается.

**Корневые причины:**

- Нет кеширования — каждый запрос MES frontend к MES API идёт напрямую в MES DB. При одновременных
  запросах от нескольких операторов нагрузка растёт линейно.
- Один инстанс MES DB обслуживает и операции записи (обновление статусов, результаты расчёта
  стоимости), и тяжёлые запросы чтения (дашборд с фильтрами) — read и write конкурируют за ресурсы.
- MES API одновременно выполняет расчёт стоимости (CPU-интенсивная операция) и обслуживает запросы
  дашборда — ресурсы одного инстанса делятся между этими задачами.
- Фильтр по статусам и пагинация не помогают, если запросы к БД остаются тяжёлыми из-за отсутствия
  индексов или неоптимальных запросов на растущем объёме данных.

### 1.6. Отсутствие наблюдаемости (observability) системы (высокая)

**Симптомы:** проблемы обнаруживаются только по жалобам клиентов; невозможно проактивно выявлять
деградацию.

**Корневые причины:**

- Нет мониторинга серверных метрик (CPU, RAM, диск, сеть) для приложений и БД.
- Нет мониторинга бизнес-метрик (количество заказов в каждом статусе, время прохождения заказа по
  этапам).
- Нет мониторинга RabbitMQ (количество сообщений в очередях, dead-letter-exchange, consumer lag).
- Нет алертинга — команда не узнаёт о проблемах, пока клиенты не пожалуются.
- Яндекс Метрика покрывает только веб-трафик магазина, но не API-трафик от B2B-партнёров и не
  внутренние системы (CRM, MES).

### 1.7. Длительный и ненадёжный процесс релиза (средняя)

**Симптомы:** задержки релизов всё увеличиваются; ручное E2E-тестирование становится узким местом.

**Корневые причины:**

- QA-инженер прогоняет E2E-сценарии вручную — это не масштабируется с ростом функциональности.
- Деплой в release и prod — ручной, нет автоматизированного rollback.
- Деплой без zero-downtime: при обновлении единственного инстанса сервис недоступен.
- Общая БД Shop DB для двух сервисов усложняет независимый деплой — миграции требуют координации.

### 1.8. Узкое место — расчёт стоимости в MES (средняя)

**Симптомы:** длительное время расчёта (2-30 минут) при растущем потоке заказов.

**Корневые причины:**

- Расчёт стоимости выполняется в MES API, который одновременно обслуживает и UI операторов, и API
  B2B-партнёров.
- Один инстанс MES API может обрабатывать ограниченное количество одновременных расчётов.
- Нет приоритизации расчётов (простые модели ждут в очереди за сложными).
- MES API читает 3D-файлы из S3 — при большом объёме файлов это дополнительная нагрузка на сеть.

## 2. Инициативы по устранению проблем

### И1. Внедрение системы наблюдаемости (мониторинг + логирование + трейсинг)

**Цель:** получить полную картину состояния системы, возможность проактивно обнаруживать проблемы и
быстро проводить расследования инцидентов.

**Состав:**

- Развёртывание Prometheus + Grafana для сбора и визуализации метрик со всех контейнеров (Shop API,
  CRM API, MES API, Shop DB, MES DB, RabbitMQ, S3).
- Внедрение централизованного логирования (OpenSearch / ELK).
- Внедрение распределённого трейсинга (OpenTelemetry + Jaeger) — особенно критично для цепочки Shop
  API → CRM API → RabbitMQ → MES API.
- Настройка алертинга на критические метрики.
- Мониторинг RabbitMQ: dead-letter-exchange, количество сообщений in flight, consumer lag.

**Приоритет: 1 (наивысший)** — без наблюдаемости невозможно эффективно диагностировать и устранять
остальные проблемы. Это фундамент для всех последующих улучшений.

### И2. Обеспечение надёжной доставки сообщений через RabbitMQ

**Цель:** устранить потерю заказов на уровне интеграции MES API ↔ CRM API.

**Состав:**

- Настройка publisher confirms и consumer acknowledgements.
- Внедрение dead-letter-exchange с мониторингом и алертингом.
- Реализация retry-политики с exponential backoff.
- Добавление идемпотентности обработки сообщений (для защиты от дублирования).
- Реализация механизма уведомления Shop API о смене статуса заказа (сейчас Shop API не подключён к
  RabbitMQ и не получает обратную связь).

**Приоритет: 2** — напрямую решает ключевую бизнес-проблему потери заказов.

### И3. Горизонтальное масштабирование и отказоустойчивость

**Цель:** устранить единую точку отказа, обеспечить возможность роста.

**Состав:**

- Добавление минимум двух инстансов для каждого приложения (Shop API, CRM API, MES API) за
  балансировщиком нагрузки (ALB/NLB).
- Настройка read-реплик для Shop DB (критично — эту БД используют и Shop API, и CRM API) и MES DB.
- Настройка кластера RabbitMQ (минимум 3 узла с кворумными очередями).
- Zero-downtime deployment (rolling update или blue-green).

**Приоритет: 3** — обеспечивает устойчивость к отказам и масштабирование при продолжающемся линейном
росте нагрузки.

### И4. Кеширование дашборда MES

**Цель:** решить проблему медленной загрузки страницы операторов.

**Состав:**

- Внедрение серверного кеша (Redis) для результатов запросов дашборда MES API → MES DB.
- Оптимизация SQL-запросов и индексов в MES DB.
- Использование read-реплик MES DB для запросов дашборда (после реализации И3).

**Приоритет: 4** — быстрый эффект для конкретной бизнес-проблемы операторов.

### И5. Внедрение API Gateway и защита публичного API

**Цель:** защитить систему от перегрузки со стороны B2B-партнёров и каскадных отказов.

**Состав:**

- Внедрение API Gateway перед MES API для трафика B2B-партнёров с rate limiting и квотами на
  партнёра.
- Разделение трафика: внутренний трафик операторов (MES frontend → MES API) и внешний B2B-трафик (
  API user → API Gateway → MES API).
- Реализация паттерна Circuit Breaker на межсервисных вызовах.
- Реализация паттерна Backpressure для управления потоком заказов через API.

**Приоритет: 5** — необходимо для стабильной работы при росте нагрузки от B2B.

### И6. Разделение Shop DB между Shop API и CRM API

**Цель:** устранить связанность через общую БД, обеспечить независимое масштабирование и деплой.

**Состав:**

- Выделение отдельной БД для CRM API с данными, специфичными для CRM (информация о продавцах,
  статусы подтверждения и т.д.).
- Синхронизация общих данных (заказы, клиенты) через API-вызовы или события в RabbitMQ вместо
  прямого доступа к чужой БД.
- Поэтапная миграция: сначала read-реплика для CRM API, затем полное разделение.

**Приоритет: 6** — стратегически важно для долгосрочной архитектуры, но требует значительного
рефакторинга.

### И7. Автоматизация тестирования и CI/CD

**Цель:** сократить время релизного цикла и повысить качество выпускаемого софта.

**Состав:**

- Автоматизация E2E-тестов (QA-инженер уже готов к этому).
- Автоматизация деплоя в release и prod (с ручным подтверждением).
- Внедрение автоматического rollback при обнаружении ошибок после деплоя.

**Приоритет: 7** — важно для долгосрочной эффективности, но не блокирует решение текущих
бизнес-проблем.

### И8. Оптимизация процесса расчёта стоимости в MES

**Цель:** сократить время ожидания расчёта для клиентов.

**Состав:**

- Горизонтальное масштабирование MES API для параллельного расчёта стоимости нескольких заказов.
- Выделение расчёта стоимости в отдельный worker-сервис, чтобы он не конкурировал с обслуживанием UI
  операторов.
- Введение приоритетных очередей (простые модели обрабатываются быстрее).
- Уведомление клиента о прогрессе расчёта (webhook или push-уведомление).

**Приоритет: 8** — улучшает UX, но не является корневой причиной потери заказов.

## 3. Приоритизация и обоснование

### Логика приоритизации

Приоритеты расставлены по принципу **«сначала видимость, потом стабильность, потом
производительность, потом архитектурное разделение»**:

1. **Без наблюдаемости (И1)** мы работаем вслепую — не можем измерить масштаб проблем, не можем
   верифицировать, что изменения помогли, не можем проактивно предотвращать инциденты.
2. **Потеря заказов (И2)** — это прямые финансовые потери и репутационный ущерб. Надёжная доставка
   сообщений решает корневую причину.
3. **Единая точка отказа (И3)** — при текущем росте (+100 заказов/месяц) и одном инстансе любой сбой
   приводит к полному простою. Особенно критично для Shop DB, которая обслуживает и магазин, и CRM.
4. **Кеширование MES (И4)** — быстрый win для бизнеса с относительно небольшими трудозатратами.
5. **API Gateway (И5)** — защита от перегрузки B2B-трафиком, изоляция внутренних пользователей.
6. **Разделение БД (И6)** — долгосрочная архитектурная инициатива, фундамент для независимости
   сервисов.
   7-8. **CI/CD и оптимизация расчёта** — важны, но могут выполняться параллельно.

### Целевая архитектура через полгода

Через шесть месяцев система «Александрит» должна выглядеть следующим образом:

- **Минимум по 2 инстанса** каждого приложения (Shop API, CRM API, MES API) за балансировщиком
  нагрузки.
- **Read-реплики** для Shop DB (обслуживает и магазин, и CRM) и MES DB; разделение нагрузки на
  чтение и запись.
- **Кластер RabbitMQ** (3 узла, кворумные очереди) с настроенными publisher confirms, consumer ack,
  dead-letter-exchange и retry-политиками.
- **Полноценная система наблюдаемости:** Prometheus + Grafana для метрик, OpenSearch для логов,
  OpenTelemetry + Jaeger для трейсинга. Настроенный алертинг.
- **Redis-кеш** для дашборда MES.
- **API Gateway** перед MES API для B2B-трафика с rate limiting и квотами.
- **Circuit Breaker** на межсервисных вызовах.
- **Автоматизированные E2E-тесты** и автоматический деплой в release/prod с ручным подтверждением.
- **Zero-downtime deployment** для всех сервисов.
- **Начат процесс разделения Shop DB** — CRM API переведён на чтение с read-реплики, спроектирована
  схема отдельной CRM DB.

### Если бы можно было выполнить только три инициативы

**1. Внедрение системы наблюдаемости (И1)**

Без наблюдаемости все остальные изменения — стрельба вслепую. Мы не знаем точных масштабов потери
заказов, не видим, где именно в цепочке Shop API → CRM API → RabbitMQ → MES API возникают задержки,
не можем измерить эффект от внесённых изменений. Мониторинг, логирование и трейсинг — это первое,
что нужно внедрить. Это даст:

- Проактивное обнаружение проблем вместо реактивного (по жалобам клиентов).
- Сокращение времени расследования инцидентов с часов/дней до минут.
- Метрики для обоснования дальнейших инвестиций в инфраструктуру перед бизнесом.
- Видимость состояния RabbitMQ — dead-letter-exchange, consumer lag, что покажет масштаб потери
  сообщений.

**2. Обеспечение надёжной доставки сообщений через RabbitMQ (И2)**

Потеря заказов — прямая потеря выручки и крупных контрактов. Корневая причина — ненадёжная
интеграция CRM API ↔ RabbitMQ ↔ MES API. Настройка гарантированной доставки, dead-letter-exchange с
мониторингом и retry-политик решает основную бизнес-проблему при относительно низких трудозатратах (
это доработка существующей интеграции, а не создание с нуля). Ожидаемый результат:

- Резкое сокращение потерянных заказов.
- Возможность отслеживать и повторно обрабатывать проблемные сообщения через dead-letter-exchange.
- Повышение доверия B2B-партнёров и сохранение контрактов.

**3. Горизонтальное масштабирование и отказоустойчивость (И3)**

С ростом +100 заказов/месяц через полгода нагрузка удвоится. Один инстанс каждого сервиса и БД —
единая точка отказа. Особенно критично: Shop DB обслуживает одновременно и Shop API, и CRM API — при
её падении недоступны оба сервиса. Добавление инстансов с балансировкой и read-реплик для БД:

- Устраняет единую точку отказа.
- Даёт запас по производительности для растущей нагрузки.
- Read-реплика Shop DB позволяет разделить нагрузку от Shop API и CRM API.
- Позволяет выполнять zero-downtime deployment.
- Кластер RabbitMQ (3 узла) повышает надёжность доставки сообщений, усиливая эффект от И2.
- Read-реплика MES DB частично решает проблему медленного дашборда MES.

### Необходимость расширения команды

Для реализации перечисленных инициатив рекомендуется нанять:

- **SRE/DevOps-инженера (второго)** — текущий DevOps-инженер один и будет перегружен задачами по
  настройке инфраструктуры мониторинга, балансировщиков, кластеризации RabbitMQ, настройке CI/CD.
  Второй инженер критичен для параллельной работы над И1 и И3.
