# Выбор и настройка мониторинга в системе

## 1. Мотивация

Текущие проблемы, которые невозможно решить без мониторинга:

1. **Потеря заказов.** Нет данных о кол-ве потерянных заказов, команда не может определить причину.
2. **Деградация производительности.** Операторы жалуются на медленную загрузку дашборда MES, но
   команда не может определить причину.
3. **Перегрузка от B2B-трафика.** Невозможно оценить, какой объём нагрузки создают B2B-партнёры и
   как он влияет на работу внутренних пользователей.
4. **Реактивный подход.** Проблемы обнаруживаются только когда клиенты жалуются или уходят.
5. **Приоритизация задач.** На основе данных о латентности и RPS можно принимать решение какие
   эндпоинты необходимо доработать/оптимизировать.
6. **Планирование масштабирования.** На основе данных мониторинга можно принимать решение о
   масштабировании.
7. **Оценка влияния изменений.** Мониторинг позволяет определить влияние релиза на
   производительность приложения.

---

## 2. Выбор подхода к мониторингу

Для разных частей системы используются разные подходы, исходя из их природы:

| Компонент                                              | Подход                 | Обоснование                                                                                                                   |
|--------------------------------------------------------|------------------------|-------------------------------------------------------------------------------------------------------------------------------|
| Shop API, CRM API, MES API (backend-сервисы)           | RED                    | RED оптимален для сервисов обрабатывающих http запросы.                                                                       |
| RabbitMQ (очередь сообщений)                           | Четыре золотых сигнала | Важно отслеживать не только ошибки, но и насыщенность (глубина очередей, consumer lag), так как это индикатор потери заказов. |
| Shop DB, MES DB, S3 (базы данных и ресурсы приложений) | USE                    | USE хорошо подходит для определения нагрузки на ресурсы.                                                                      |

## 3. Метрики и их описание

### 3.1. RabbitMQ - «Четыре золотых сигнала»

1. Number of dead-letter-exchange letters in RabbitMQ
    * Зачем: Прямой индикатор потери заказов.
    * Ярлыки:
        - `queue` - имя очереди.
        - `reason` - причина попадания в DLX.

2. Number of messages in flight in RabbitMQ
    * Зачем: показывает кол-во необработанных сообщений. Может служить обоснованием для внедрения
      backpressure / circuit breaker.
    * Ярлыки:
        - `queue` - имя очереди.
        - `consumer` - идентификатор consumer-а.

### 3.2. API-сервисы - подход RED

1. Number of requests (RPS) for Shop API / CRM API / MES API
    * Зачем: базовая метрика нагрузки. Позволяет отслеживать тренд роста, обнаруживать всплески,
      сравнивать нагрузку между сервисами.
    * Ярлыки:
        - `method` - HTTP-метод (`GET`, `POST`, `PUT`).
        - `endpoint` - путь запроса.
        - `status_code` - HTTP-код ответа.

2. Number of requests (RPS) per user for Shop API / CRM API / MES API
    * Зачем: определение какой B2B-партнёр или внутренний сервис создаёт наибольшую нагрузку.
    * Ярлыки:
        - `user_id` - идентификатор пользователя или B2B-партнёра.
        - `endpoint` - путь запроса.

3. Response time (latency) for Shop API / CRM API / MES API
    * Зачем: ключевая метрика пользовательского опыта. Признак роста деградации произв-ти.
    * Ярлыки:
        - `endpoint` - путь запроса (выделение медленных эндпоинтов).
        - `method` - HTTP-метод.

4. Number of HTTP 200 / HTTP 500 for Shop API / CRM API / MES API
    * Зачем: error rate - ключевой сигнал RED-подхода. Рост HTTP 500 - индикатор сбоев на
      бэкенде.
    * Ярлыки:
        - `endpoint` - путь запроса.

### 3.3. Базы данных - подход USE

1. Memory Utilisation for Shop DB / MES DB
    * Зачем: высокое потребление ведёт к росту I/O и деградации запросов.
    * Ярлыки:
        - `instance` - идентификатор инстанса БД.

2. Number of connections for Shop DB / MES DB
    * Зачем: postgreSQL имеет ограничение в кол-ве соединений.
    * Ярлыки:
        - `instance` - идентификатор инстанса БД.
        - `state` - состояние соединения (`active`, `idle`, `idle_in_transaction`).
        - `application_name` - имя приложения (`shop_api`, `crm_api`).

3. Size of Shop DB / MES DB
    * Зачем: для планирования расширения диска и контроля роста данных.
    * Ярлыки:
        - `instance` - идентификатор инстанса БД.

### 3.4. Ресурсы приложений - подход USE

1. CPU % for Shop API / CRM API / MES API
    * Зачем: базовый индикатор нагрузки.
    * Ярлыки:
        - `instance` - идентификатор EC2-инстанса.
        - `service` - имя сервиса.

2. Memory Utilisation for Shop API / CRM API / MES API
    * Зачем: базовый индикатор нагрузки.
    * Ярлыки:
        - `instance` - идентификатор EC2-инстанса.
        - `service` - имя сервиса.

### 3.5. S3 - подход USE

1. Size of S3 storage
    * Зачем: для планирования расширения диска и контроля роста данных.
    * Ярлыки:
        - `bucket` - имя бакета.

### 3.6. Трафик (сетевые метрики)

1. Kb transferred (received) for Shop API / MES API
    * Зачем: мониторинг объема данных; обнаружение аномалий.
    * Ярлыки:
        - `service` - имя сервиса.
        - `endpoint` - путь запроса.

2. Kb provided (sent) for Shop API / MES API
    * Зачем: мониторинг объема данных; обнаружение аномалий.
    * Ярлыки:
        - `service` - имя сервиса.
        - `endpoint` - путь запроса.

### 3.7. Сессии

Number of simultaneous sessions for Shop API / MES API

* Зачем: кол-во пользо-ей в каждый момент времени. Позволит мотивировать масштабирование.
* Ярлыки:
    - `instance` - идентификатор инстанса.

### 3.8. Бизнес-метрики

Количество заказов по статусам

* Зачем: отслеживание накоплений заказов в определенных статусах.
* Ярлыки:
    - `status` - статус заказа.
    - `source` - источник заказа (`b2c`, `b2b`).

## 4. План действий

### Этап 1. Инфраструктура мониторинга (неделя 1-2)

1. Развернуть Prometheus.
2. Развернуть Grafana.
3. Настроить Alertmanager.
4. Настроить retention policy для хранения метрик.

### Этап 2. Сбор метрик приложений (неделя 2-3)

1. Подключить Prometheus-экспортеры к Shop API и CRM API.
2. Подключить Prometheus-экспортер к MES API.
3. Добавить кастомные бизнес-метрики (заказы по статусам).

### Этап 3. Сбор метрик инфраструктуры (неделя 3-4)

1. Настроить мониторинг БД.
2. Настроить мониторинг RabbitMQ.
3. Настроить мониторинг EC2-инстансов.
4. Настроить мониторинг S3.

### Этап 4. Дашборды и алертинг (неделя 4-5)

1. Создать дашборды в Grafana.
2. Настроить алерты по ключевым метрикам.
