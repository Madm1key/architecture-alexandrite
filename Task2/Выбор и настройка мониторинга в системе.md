# Выбор и настройка мониторинга в системе

## 1. Мотивация

Текущие проблемы, которые невозможно решить без мониторинга:

1. **Потеря заказов.** Нет данных о кол-ве потерянных заказов, команда не может определить причину.
2. **Деградация производительности.** Операторы жалуются на медленную загрузку дашборда MES, но
   команда не может определить причину.
3. **Перегрузка от B2B-трафика.** Невозможно оценить, какой объём нагрузки создают B2B-партнёры и
   как он влияет на работу внутренних пользователей.
4. **Реактивный подход.** Проблемы обнаруживаются только когда клиенты жалуются или уходят.

---

## 2. Выбор подхода к мониторингу

Для разных частей системы используются разные подходы, исходя из их природы:

| Компонент                                    | Подход                 | Обоснование                                                                                                                   |
|----------------------------------------------|------------------------|-------------------------------------------------------------------------------------------------------------------------------|
| Shop API, CRM API, MES API (backend-сервисы) | RED                    | RED оптимален для сервисов обрабатывающих http запросы.                                                                       |
| RabbitMQ (очередь сообщений)                 | Четыре золотых сигнала | Важно отслеживать не только ошибки, но и насыщенность (глубина очередей, consumer lag), так как это индикатор потери заказов. |
| Shop DB, MES DB, S3 (базы данных)            | USE                    | USE хорошо подходит для определения нагрузки на ресурсы.                                                                      |

---

## 3. Метрики и их описание

### 3.1. RabbitMQ - «Четыре золотых сигнала»

1. Number of dead-letter-exchange letters in RabbitMQ
    * Зачем: Прямой индикатор потери заказов.
    * Ярлыки:
        - `queue` - имя очереди.
        - `reason` - причина попадания в DLX.

2. Number of messages in flight in RabbitMQ
    * Зачем: показывает кол-во необработанных сообщений. Может служить обоснованием для внедрения
      backpressure / circuit breaker.
    * Ярлыки:
        - `queue` - имя очереди.
        - `consumer` - идентификатор consumer-а.

### 3.2. API-сервисы - подход RED

1. Number of requests (RPS) for Shop API / CRM API / MES API
    * Зачем: базовая метрика нагрузки. Позволяет отслеживать тренд роста, обнаруживать всплески,
      сравнивать нагрузку между сервисами.
    * Ярлыки:
        - `method` - HTTP-метод (`GET`, `POST`, `PUT`).
        - `endpoint` - путь запроса.
        - `status_code` - HTTP-код ответа.

2. Number of requests (RPS) per user for Shop API / MES API
    * Зачем: какой B2B-партнёр создаёт наибольшую нагрузку. Обоснование для введения rate limiting.
    * Ярлыки:
        - `user_id` - идентификатор пользователя или B2B-партнёра.
        - `endpoint` - путь запроса.

3. Response time (latency) for Shop API / CRM API / MES API
    * Зачем: Ключевая метрика пользовательского опыта. Рост latency - первый признак деградации.
      Для MES API критично: медленная загрузка дашборда - основная жалоба операторов. Для Shop API
      влияет на конверсию.
    * Ярлыки:
        - `endpoint` - путь запроса (выделение медленных эндпоинтов).
        - `method` - HTTP-метод.
        - `quantile` - перцентили (p50, p95, p99).

4. Number of HTTP 200 / HTTP 500 for Shop API / CRM API / MES API
    * Зачем: Error rate - ключевой сигнал RED-подхода. Рост HTTP 500 - индикатор сбоев на
      бэкенде. Алерт на резкий рост позволит мгновенно обнаружить сбой после деплоя или при
      перегрузке.
    * Ярлыки:
        - `endpoint` - путь запроса.
        - `error_type` - тип ошибки (`database_error`, `timeout`, `queue_error` и т.д.).

### 3.3. Базы данных - подход USE

1. Memory Utilisation for Shop DB / MES DB
    * Зачем: PostgreSQL активно использует RAM для кеширования (shared_buffers, OS page cache).
      Высокое потребление ведёт к росту I/O и деградации запросов. Для Shop DB критично вдвойне -
      её используют и Shop API, и CRM API.
    * Ярлыки:
        - `instance` - идентификатор инстанса БД.
        - `type` - тип памяти (`shared_buffers`, `work_mem`, `total`).

2. Number of connections for Shop DB / MES DB
    * Зачем: PostgreSQL имеет ограничение `max_connections`. Исчерпание пула - отказ в
      обслуживании. Для Shop DB: два сервиса (Shop API + CRM API) конкурируют за один пул - важно
      видеть, кто потребляет больше.
    * Ярлыки:
        - `instance` - идентификатор инстанса БД.
        - `state` - состояние соединения (`active`, `idle`, `idle_in_transaction`).
        - `application_name` - имя приложения (`shop_api`, `crm_api`).

3. Size of Shop DB / MES DB
    * Зачем: Мониторинг роста данных для capacity planning. При +100 заказов/месяц БД будут
      расти предсказуемо - важно заранее планировать расширение диска и архивацию.
    * Ярлыки:
        - `instance` - идентификатор инстанса БД.
        - `database` - имя базы данных.

### 3.4. Ресурсы приложений - подход USE

1. CPU % for Shop API / CRM API / MES API
    * Зачем: Базовый индикатор нагрузки. Для MES API особенно критично: расчёт стоимости
      3D-моделей - CPU-интенсивная операция, конкурирующая с обработкой запросов дашборда и API.
    * Ярлыки:
        - `instance` - идентификатор EC2-инстанса.
        - `service` - имя сервиса.

2. Memory Utilisation for Shop API / CRM API / MES API
    * Зачем: Утечки памяти и рост heap - частые причины деградации Java (Spring Boot) и .NET
      приложений. Позволяет обнаружить утечку до падения сервиса с OOM.
    * Ярлыки:
        - `instance` - идентификатор EC2-инстанса.
        - `service` - имя сервиса.
        - `type` - тип памяти (`heap`, `non_heap`, `total` для JVM; `managed_heap`, `total` для
          .NET).

### 3.5. S3 - подход USE

1. Size of S3 storage
    * Зачем: 3D-файлы - тяжёлые объекты. При росте заказов хранилище растёт. Нужно для capacity
      planning и контроля затрат (S3 тарифицируется по объёму).
    * Ярлыки:
        - `bucket` - имя бакета.

### 3.6. Трафик (сетевые метрики)

1. Kb transferred (received) for Shop API / MES API
    * Зачем: Shop API - обнаружение аномалий при загрузке пользователями 3D-файлов. MES API -
      мониторинг объёма данных от B2B-партнёров, выявление партнёров с аномально большим трафиком.
    * Ярлыки:
        - `service` - имя сервиса.
        - `endpoint` - путь запроса.

2. Kb provided (sent) for Shop API / MES API
    * Зачем: Shop API - выявление неоптимальных ответов API (отсутствие пагинации, избыточные
      данные). MES API - объём данных дашборда для операторов и ответов B2B-партнёрам.
    * Ярлыки:
        - `service` - имя сервиса.
        - `endpoint` - путь запроса.

   > Примечание: трафик CRM API можно не внедрять на первом этапе - CRM используется только
   > внутренними пользователями, объём трафика контролируем и предсказуем.

### 3.7. Сессии

1. Number of simultaneous sessions for Shop API
    * Зачем: Показывает реальную нагрузку от пользователей в каждый момент. Совместно с RPS и
      latency позволяет оценить capacity и пороги масштабирования.
    * Ярлыки:
        - `instance` - идентификатор инстанса.

2. Number of simultaneous sessions for MES API
    * Зачем: Суммарные сессии от операторов и B2B-партнёров. Критично для понимания, как
      B2B-трафик влияет на работу операторов.
    * Ярлыки:
        - `instance` - идентификатор инстанса.
        - `source` - источник сессии (`internal` для операторов, `external` для B2B).

### 3.8. Дополнительные метрики (не из исходного списка)

1. Количество заказов по статусам
    * Зачем: Аномальное накопление заказов в определённом статусе (например, `SUBMITTED` без
      перехода в `PRICE_CALCULATED`) - прямой индикатор проблемы в цепочке обработки.
    * Ярлыки:
        - `status` - статус заказа (`INITIATED`, `FILE_UPLOADED`, `SUBMITTED`, `PRICE_CALCULATED`,
          `MANUFACTURING_APPROVED`, `MANUFACTURING_STARTED`, `MANUFACTURING_COMPLETED`, `PACKAGING`,
          `SHIPPED`, `CLOSED`).
        - `source` - источник заказа (`b2c`, `b2b`).

2. Среднее время нахождения заказа в каждом статусе
    * Зачем: Обнаружение «бутылочных горлышек». Рост времени `SUBMITTED` → `PRICE_CALCULATED` -
      индикатор перегрузки MES API расчётами. Рост `PRICE_CALCULATED` → `MANUFACTURING_APPROVED` -
      проблема на стороне CRM/продавцов.
    * Ярлыки:
        - `from_status` - исходный статус.
        - `to_status` - целевой статус.
        - `source` - источник заказа (`b2c`, `b2b`).

## 4. План действий

### Этап 1. Инфраструктура мониторинга (неделя 1-2)

1. **Развернуть Prometheus** - создать EC2-инстанс (или использовать Managed Prometheus, если
   доступен в Yandex Cloud) для сбора метрик со всех сервисов.
2. **Развернуть Grafana** - создать инстанс для визуализации метрик, подключить Prometheus как data
   source.
3. **Настроить Alertmanager** - развернуть рядом с Prometheus для отправки алертов в выбранный
   канал (Telegram, Slack, email).
4. **Настроить хранение метрик** - определить retention policy (минимум 30 дней полной детализации,
   1 год агрегированных данных).

### Этап 2. Сбор метрик приложений (неделя 2-3)

5. **Добавить Prometheus-экспортеры в Shop API и CRM API** - подключить
   `micrometer-registry-prometheus` (стандарт для Spring Boot). Это даст метрики HTTP-запросов (RPS,
   latency, status codes), JVM (heap, GC, threads), connection pool.
6. **Добавить Prometheus-экспортер в MES API** - подключить `prometheus-net` (стандарт для .NET/C#).
   Аналогичный набор метрик.
7. **Добавить кастомные бизнес-метрики** - инструментировать код для отслеживания количества заказов
   по статусам и времени переходов между статусами.

### Этап 3. Сбор метрик инфраструктуры (неделя 3-4)

8. **Настроить мониторинг PostgreSQL** - установить `postgres_exporter` для Shop DB и MES DB.
   Метрики: connections, query duration, replication lag (на будущее), размер БД.
9. **Настроить мониторинг RabbitMQ** - включить встроенный `rabbitmq_prometheus` плагин. Метрики:
   queue depth, in-flight messages, dead-letter-exchange, consumers count, publish/consume rates.
10. **Настроить мониторинг EC2-инстансов** - установить `node_exporter` на каждый инстанс. Метрики:
    CPU, RAM, disk I/O, network.
11. **Настроить мониторинг S3** - использовать CloudWatch-метрики Yandex Cloud (или аналог) для
    объёма хранилища и количества запросов.

### Этап 4. Дашборды и алертинг (неделя 4-5)

12. **Создать дашборды в Grafana:**
    - **Overview** - общая картина системы: RPS, error rate, latency всех сервисов на одном экране.
    - **Per-service** - детальный дашборд для каждого сервиса (Shop API, CRM API, MES API).
    - **Databases** - состояние Shop DB и MES DB: connections, query latency, размер.
    - **RabbitMQ** - глубина очередей, DLX, consumer lag, publish/consume rate.
    - **Business metrics** - заказы по статусам, время обработки, источник (B2C/B2B).
13. **Настроить алерты:**
    - Error rate > 5% на любом API - severity: critical.
    - Latency p99 > 5s на Shop API / CRM API - severity: warning.
    - Latency p99 > 10s на MES API (дашборд) - severity: warning.
    - DLX messages > 0 - severity: critical (каждое сообщение в DLX - потенциально потерянный
      заказ).
    - RabbitMQ consumers count = 0 на любой очереди - severity: critical.
    - CPU > 80% на любом инстансе - severity: warning.
    - DB connections > 80% от max_connections - severity: warning.
    - Disk usage > 80% - severity: warning.

## 5. Показатели насыщенности (дополнительное задание)

Насыщенность (Saturation) - это степень использования ресурса, при которой начинается деградация.
Ниже описаны пороговые значения и действия при их превышении.

### 5.1. CPU на EC2-инстансах (Shop API, CRM API, MES API)

| Порог    | Значение | Обоснование                                                                                   |
|----------|----------|-----------------------------------------------------------------------------------------------|
| Warning  | **70%**  | Приложение начинает конкурировать с ОС за ресурсы. При всплесках нагрузки запас отсутствует.  |
| Critical | **85%**  | Высокая вероятность деградации latency. При 85%+ CPU увеличивается время ожидания в runqueue. |

**Действия при превышении:**

- Warning: автоматический алерт в канал дежурного, создание тикета на анализ причин.
- Critical: алерт + звонок дежурному (PagerDuty/VictorOps). Немедленное расследование. При
  сохранении > 15 минут - добавление инстанса (горизонтальное масштабирование).

### 5.2. Memory Utilisation на EC2-инстансах

| Порог    | Значение | Обоснование                                                                                                                  |
|----------|----------|------------------------------------------------------------------------------------------------------------------------------|
| Warning  | **75%**  | Для JVM (Shop API, CRM API): GC начинает работать агрессивнее, увеличивая паузы. Для .NET (MES API): аналогичная деградация. |
| Critical | **90%**  | Высокий риск OOM. ОС может начать использовать swap, что катастрофически снижает производительность.                         |

**Действия при превышении:**

- Warning: создание тикета, анализ heap dump, проверка на утечки памяти.
- Critical: алерт + звонок дежурному. Рестарт сервиса, если утечка подтверждена. Увеличение RAM или
  добавление инстанса.

### 5.3. Database connections (Shop DB, MES DB)

| Порог    | Значение                     | Обоснование                                                                                       |
|----------|------------------------------|---------------------------------------------------------------------------------------------------|
| Warning  | **70%** от `max_connections` | Запас для пиковых нагрузок. При 70% новые запросы могут начать ожидать в очереди пула соединений. |
| Critical | **85%** от `max_connections` | При исчерпании пула - отказ в обслуживании для новых запросов.                                    |

**Действия при превышении:**

- Warning: создание тикета, анализ idle и idle_in_transaction соединений. Для Shop DB: проверить
  баланс соединений между Shop API и CRM API.
- Critical: алерт + звонок дежурному. Принудительное закрытие idle-in-transaction соединений.
  Увеличение `max_connections` или внедрение pgBouncer (connection pooler).

### 5.4. RabbitMQ - глубина очередей

| Порог    | Значение                     | Обоснование                                                                                                                |
|----------|------------------------------|----------------------------------------------------------------------------------------------------------------------------|
| Warning  | **1000 сообщений** в очереди | Consumer-ы не успевают обрабатывать. При текущей нагрузке очередь не должна содержать более нескольких десятков сообщений. |
| Critical | **5000 сообщений** в очереди | Consumer-ы серьёзно отстают. Высокий риск потери заказов при переполнении или таймауте.                                    |

**Действия при превышении:**

- Warning: алерт дежурному, проверка состояния consumer-ов (MES API, CRM API).
- Critical: алерт + звонок, немедленная проверка consumer-ов. Если consumer-ы работают - увеличить
  количество consumer-ов (масштабирование сервиса). Если consumer упал - перезапуск.

### 5.5. Disk usage (БД, RabbitMQ)

| Порог    | Значение | Обоснование                                                                                                                |
|----------|----------|----------------------------------------------------------------------------------------------------------------------------|
| Warning  | **70%**  | При текущем линейном росте (+100 заказов/месяц) диск будет заполняться предсказуемо. 70% - время для планового расширения. |
| Critical | **85%**  | PostgreSQL может перестать принимать записи при заполнении диска. RabbitMQ начнёт сбрасывать сообщения.                    |

**Действия при превышении:**

- Warning: создание тикета на расширение диска. Анализ возможности архивации старых данных.
- Critical: алерт + звонок дежурному. Экстренное расширение диска через Yandex Cloud console.
  Удаление старых логов/temp-файлов как временная мера.

### 5.6. Latency p99

| Порог                      | Значение (p99) | Обоснование                                                                     |
|----------------------------|----------------|---------------------------------------------------------------------------------|
| Warning (Shop API)         | **3 секунды**  | Пользователь онлайн-магазина ожидает быструю загрузку. > 3s - потеря конверсии. |
| Warning (MES API, дашборд) | **5 секунд**   | Операторы уже жалуются на медленную загрузку.                                   |
| Critical (любой API)       | **10 секунд**  | Фактическая неработоспособность для пользователя.                               |

**Действия при превышении:**

- Warning: создание тикета, анализ медленных запросов (slow query log в БД, profiling API).
- Critical: алерт + звонок. Немедленный анализ: проверка CPU/RAM, состояния БД, глубины очередей
  RabbitMQ. Рассмотрение экстренного масштабирования.
