# Выбор и настройка мониторинга в системе

## 1. Мотивация

### Почему нужен мониторинг

Сейчас единственный источник информации о состоянии системы «Александрит» — Яндекс Метрика, которая покрывает только веб-трафик онлайн-магазина. После открытия публичного API для B2B-партнёров значительная часть нагрузки идёт в обход веб-интерфейса напрямую в MES API, и этот трафик полностью невидим для команды.

Текущие проблемы, которые невозможно решить без мониторинга:

1. **Потеря заказов.** Команда не знает масштаб проблемы — о потерянных заказах узнают только по жалобам клиентов. Нет данных о количестве сообщений в dead-letter-exchange RabbitMQ, нет видимости consumer lag.
2. **Деградация производительности.** Операторы жалуются на медленную загрузку дашборда MES, но команда не может определить корневую причину — нет данных о latency запросов, загрузке CPU/RAM, количестве соединений к БД.
3. **Перегрузка от B2B-трафика.** Невозможно оценить, какой объём нагрузки создают B2B-партнёры и как он влияет на работу внутренних пользователей (операторов, продавцов).
4. **Реактивный подход.** Проблемы обнаруживаются только когда клиенты жалуются или уходят. Нет возможности предотвращать инциденты заранее.

### Что мониторинг даст компании

- **Сокращение времени обнаружения инцидентов** — с часов/дней (по жалобам клиентов) до минут (по алертам).
- **Обоснование инвестиций в инфраструктуру** — метрики покажут бизнесу, где именно узкие места и как они влияют на конверсию и выручку.
- **Сохранение B2B-контрактов** — возможность отслеживать SLA для партнёров и проактивно реагировать на деградацию.
- **Оптимизация ресурсов** — данные о загрузке покажут, где ресурсов не хватает, а где они расходуются впустую.
- **Capacity planning** — на основе трендов роста нагрузки можно планировать масштабирование заранее.

## 2. Выбор подхода к мониторингу

Для разных частей системы используются разные подходы, исходя из их природы:

| Компонент | Подход | Обоснование |
|---|---|---|
| Shop API, CRM API, MES API (backend-сервисы) | **RED** (Rate, Errors, Duration) | Сервисы обрабатывают запросы — RED оптимален для request-driven систем. Даёт прямую видимость пользовательского опыта. |
| RabbitMQ (очередь сообщений) | **«Четыре золотых сигнала»** (Latency, Traffic, Errors, Saturation) | Очередь — центральный элемент интеграции. Важно отслеживать не только ошибки, но и насыщенность (глубина очередей, consumer lag), так как это индикатор потери заказов. |
| Shop DB, MES DB (базы данных) | **USE** (Utilization, Saturation, Errors) | БД — это ресурсы. USE показывает, насколько они загружены, есть ли очередь ожидания и возникают ли ошибки. Критично для Shop DB, которую используют и Shop API, и CRM API. |
| S3 (хранилище 3D-файлов) | **USE** (Utilization, Saturation, Errors) | Хранилище — ресурс с конечной ёмкостью. Важно следить за объёмом и ошибками доступа. |

## 3. Метрики и их описание

### 3.1. RabbitMQ — «Четыре золотых сигнала»

#### Метрика 1. Number of `dead-letter-exchange` letters in RabbitMQ

**Зачем:** dead-letter-exchange — это прямой индикатор потери заказов. Сообщения попадают в DLX, когда consumer не может их обработать (reject, timeout, TTL). Каждое необработанное сообщение — это потенциально потерянный заказ. Это одна из самых критичных метрик для бизнеса.

**Ярлыки:**
- `queue` — имя исходной очереди (для идентификации, на каком этапе произошла ошибка: создание заказа, расчёт стоимости, смена статуса).
- `reason` — причина попадания в DLX (`rejected`, `expired`, `maxlen`).

#### Метрика 2. Number of messages `in flight` in RabbitMQ

**Зачем:** показывает количество сообщений, переданных consumer-у, но ещё не подтверждённых (unacked). Рост этой метрики означает, что consumer-ы не успевают обрабатывать сообщения — либо перегружены, либо зависли. Высокий in flight при растущей нагрузке — предвестник потери заказов.

**Ярлыки:**
- `queue` — имя очереди (для разделения потоков между CRM API и MES API).
- `consumer` — идентификатор consumer-а.

### 3.2. API-сервисы — подход RED

#### Метрики 3, 4, 5. Number of requests (RPS) for Shop API / CRM API / MES API

**Зачем:** базовая метрика нагрузки на каждый сервис. Позволяет отслеживать тренд роста, обнаруживать аномальные всплески (возможные DDoS или массовые retry от партнёров), сравнивать нагрузку между сервисами.

**Ярлыки:**
- `method` — HTTP-метод (`GET`, `POST`, `PUT`).
- `endpoint` — путь запроса (например, `/api/orders`, `/api/orders/{id}/status`).
- `status_code` — HTTP-код ответа (для быстрой фильтрации по успешным/неуспешным запросам).

#### Метрики 6, 8. Number of requests (RPS) per user for Shop API / MES API

**Зачем:**
- *Shop API (метрика 6):* выявление аномального поведения пользователей — возможные боты или DDoS.
- *MES API (метрика 8):* критично для B2B — позволяет определить, какой партнёр создаёт наибольшую нагрузку, и обосновать введение rate limiting и квот. Сейчас API user подключается напрямую к MES API без gateway — эта метрика покажет масштаб проблемы.

**Ярлыки:**
- `user_id` / `partner_id` — идентификатор пользователя или B2B-партнёра.
- `endpoint` — путь запроса.

> **Примечание:** метрику 7 (RPS per user для CRM API) можно не внедрять на первом этапе — CRM используется только внутренними продавцами, количество пользователей ограничено и контролируемо.

#### Метрики 19, 20, 21. Response time (latency) for Shop API / CRM API / MES API

**Зачем:** ключевая метрика пользовательского опыта. Рост latency — первый признак деградации. Для MES API особенно важно, так как медленная загрузка дашборда — одна из основных жалоб операторов. Для Shop API — влияет на конверсию онлайн-магазина.

**Ярлыки:**
- `endpoint` — путь запроса (критично для выделения медленных эндпоинтов, например, дашборд MES vs. обновление статуса).
- `method` — HTTP-метод.
- `quantile` — перцентили (p50, p95, p99) для понимания не только среднего, но и хвоста распределения.

#### Метрики 25-30. Number of HTTP 200 / HTTP 500 for Shop API / CRM API / MES API

**Зачем:** Error rate — ключевой сигнал RED-подхода. Рост HTTP 500 — индикатор сбоев на бэкенде. Соотношение HTTP 200 к общему числу запросов показывает Success Rate. Алерт на резкий рост 500-х позволит мгновенно обнаружить сбой после деплоя или при перегрузке.

**Ярлыки:**
- `endpoint` — путь запроса.
- `error_type` — тип ошибки (для HTTP 500: `database_error`, `timeout`, `queue_error` и т.д.), если приложение умеет его классифицировать.

> **Примечание:** метрику 31 (дубликат HTTP 500 для Shop API) считаю опечаткой в исходном списке — покрывается метрикой 28.

### 3.3. Базы данных — подход USE

#### Метрики 15, 16. Memory Utilisation for Shop DB / MES DB instance

**Зачем:** PostgreSQL активно использует RAM для кеширования страниц (shared_buffers, OS page cache). Высокое потребление памяти при растущем объёме данных приводит к увеличению I/O и деградации запросов. Для Shop DB критично вдвойне — её используют и Shop API, и CRM API.

**Ярлыки:**
- `instance` — идентификатор инстанса БД.
- `type` — тип памяти (`shared_buffers`, `work_mem`, `total`).

#### Метрики 17, 18. Number of connections for Shop DB / MES DB instance

**Зачем:** количество соединений — прямой индикатор нагрузки на БД. PostgreSQL имеет ограничение `max_connections`. Исчерпание пула соединений приводит к отказу в обслуживании. Для Shop DB: два сервиса (Shop API + CRM API) конкурируют за один пул — важно видеть, кто потребляет больше.

**Ярлыки:**
- `instance` — идентификатор инстанса БД.
- `state` — состояние соединения (`active`, `idle`, `idle_in_transaction`).
- `application_name` — имя приложения (`shop_api`, `crm_api`) — критично для Shop DB, чтобы различать нагрузку от двух сервисов.

#### Метрики 23, 24. Size of Shop DB / MES DB instance

**Зачем:** мониторинг роста объёма данных для capacity planning. При линейном росте заказов (+100/месяц) БД будут расти. Важно заранее планировать расширение дискового пространства и оценивать необходимость архивации старых данных.

**Ярлыки:**
- `instance` — идентификатор инстанса БД.
- `database` — имя базы данных.

### 3.4. Ресурсы приложений — подход USE

#### Метрики 9, 10, 11. CPU % for Shop API / CRM API / MES API

**Зачем:** CPU — базовый индикатор нагрузки. Для MES API особенно критично: расчёт стоимости 3D-моделей — CPU-интенсивная операция, конкурирующая с обработкой запросов дашборда и API. Высокий CPU покажет необходимость горизонтального масштабирования или выделения расчёта в отдельный worker.

**Ярлыки:**
- `instance` — идентификатор EC2-инстанса.
- `service` — имя сервиса.

#### Метрики 12, 13, 14. Memory Utilisation for Shop API / CRM API / MES API

**Зачем:** утечки памяти, рост heap — частые причины деградации Java-приложений (Shop API, CRM API на Spring Boot) и .NET-приложений (MES API на C#). Мониторинг позволяет обнаружить утечки до того, как сервис упадёт с OOM.

**Ярлыки:**
- `instance` — идентификатор EC2-инстанса.
- `service` — имя сервиса.
- `type` — тип памяти (`heap`, `non_heap`, `total` — для JVM; `managed_heap`, `total` — для .NET).

### 3.5. S3 — подход USE

#### Метрика 22. Size of S3 storage

**Зачем:** 3D-файлы — это тяжёлые объекты. При росте заказов хранилище будет расти. Нужно отслеживать для capacity planning и принятия решений об архивации/удалении. Также важно для контроля затрат (S3 тарифицируется по объёму).

**Ярлыки:**
- `bucket` — имя бакета.

### 3.6. Трафик (сетевые метрики)

#### Метрики 34, 37. Kb transferred (received) for Shop API / MES API

**Зачем:**
- *Shop API:* пользователи загружают 3D-файлы, которые могут быть значительного размера. Мониторинг входящего трафика позволяет обнаружить аномалии (например, загрузку чрезмерно больших файлов).
- *MES API:* B2B-партнёры отправляют 3D-модели через API. Мониторинг покажет объём данных от B2B и поможет выявить партнёров с аномально большим трафиком.

**Ярлыки:**
- `service` — имя сервиса.
- `endpoint` — путь запроса.

#### Метрики 38, 40. Kb provided (sent) for Shop API / MES API

**Зачем:**
- *Shop API:* объём данных, отдаваемых клиентам (списки товаров, изображения). Рост может указывать на неоптимальные ответы API (отсутствие пагинации, избыточные данные).
- *MES API:* объём данных дашборда для операторов и ответов B2B-партнёрам.

**Ярлыки:**
- `service` — имя сервиса.
- `endpoint` — путь запроса.

> **Примечание:** метрики 35, 36, 39 (трафик CRM API) можно не внедрять на первом этапе — CRM используется только внутренними пользователями, объём трафика контролируем и предсказуем.

### 3.7. Сессии

#### Метрика 31 (32). Number of simultaneous sessions for Shop API

**Зачем:** количество одновременных сессий онлайн-магазина показывает реальную нагрузку от пользователей в каждый момент времени. Совместно с RPS и latency позволяет оценить capacity и определить пороги для масштабирования.

**Ярлыки:**
- `instance` — идентификатор инстанса.

#### Метрика 34 (из списка 33). Number of simultaneous sessions for MES API

**Зачем:** количество одновременных сессий MES API — суммарно от операторов (через MES frontend) и B2B-партнёров (через API). Критично для понимания, как B2B-трафик влияет на работу операторов.

**Ярлыки:**
- `instance` — идентификатор инстанса.
- `source` — источник сессии (`internal` для операторов, `external` для B2B).

### 3.8. Дополнительные метрики (не из исходного списка)

#### Метрика: Количество заказов по статусам

**Зачем:** бизнес-метрика, которая показывает распределение заказов по этапам обработки. Аномальное накопление заказов в определённом статусе (например, `SUBMITTED` без перехода в `PRICE_CALCULATED`) — прямой индикатор проблемы в цепочке обработки.

**Ярлыки:**
- `status` — статус заказа (`INITIATED`, `FILE_UPLOADED`, `SUBMITTED`, `PRICE_CALCULATED`, `MANUFACTURING_APPROVED`, `MANUFACTURING_STARTED`, `MANUFACTURING_COMPLETED`, `PACKAGING`, `SHIPPED`, `CLOSED`).
- `source` — источник заказа (`b2c`, `b2b`).

#### Метрика: Среднее время нахождения заказа в каждом статусе

**Зачем:** позволяет обнаружить «бутылочные горлышки» в бизнес-процессе. Если среднее время в статусе `SUBMITTED` → `PRICE_CALCULATED` растёт — это индикатор перегрузки MES API расчётами. Если растёт время `PRICE_CALCULATED` → `MANUFACTURING_APPROVED` — проблема на стороне CRM/продавцов.

**Ярлыки:**
- `from_status` — исходный статус.
- `to_status` — целевой статус.
- `source` — источник заказа (`b2c`, `b2b`).

#### Метрика: Количество consumer-ов RabbitMQ

**Зачем:** если consumer отключился (упал сервис) — сообщения будут накапливаться в очереди. Метрика позволяет сразу обнаружить отключение consumer-а и среагировать до того, как очередь переполнится.

**Ярлыки:**
- `queue` — имя очереди.

## 4. План действий

### Этап 1. Инфраструктура мониторинга (неделя 1-2)

1. **Развернуть Prometheus** — создать EC2-инстанс (или использовать Managed Prometheus, если доступен в Yandex Cloud) для сбора метрик со всех сервисов.
2. **Развернуть Grafana** — создать инстанс для визуализации метрик, подключить Prometheus как data source.
3. **Настроить Alertmanager** — развернуть рядом с Prometheus для отправки алертов в выбранный канал (Telegram, Slack, email).
4. **Настроить хранение метрик** — определить retention policy (минимум 30 дней полной детализации, 1 год агрегированных данных).

### Этап 2. Сбор метрик приложений (неделя 2-3)

5. **Добавить Prometheus-экспортеры в Shop API и CRM API** — подключить `micrometer-registry-prometheus` (стандарт для Spring Boot). Это даст метрики HTTP-запросов (RPS, latency, status codes), JVM (heap, GC, threads), connection pool.
6. **Добавить Prometheus-экспортер в MES API** — подключить `prometheus-net` (стандарт для .NET/C#). Аналогичный набор метрик.
7. **Добавить кастомные бизнес-метрики** — инструментировать код для отслеживания количества заказов по статусам и времени переходов между статусами.

### Этап 3. Сбор метрик инфраструктуры (неделя 3-4)

8. **Настроить мониторинг PostgreSQL** — установить `postgres_exporter` для Shop DB и MES DB. Метрики: connections, query duration, replication lag (на будущее), размер БД.
9. **Настроить мониторинг RabbitMQ** — включить встроенный `rabbitmq_prometheus` плагин. Метрики: queue depth, in-flight messages, dead-letter-exchange, consumers count, publish/consume rates.
10. **Настроить мониторинг EC2-инстансов** — установить `node_exporter` на каждый инстанс. Метрики: CPU, RAM, disk I/O, network.
11. **Настроить мониторинг S3** — использовать CloudWatch-метрики Yandex Cloud (или аналог) для объёма хранилища и количества запросов.

### Этап 4. Дашборды и алертинг (неделя 4-5)

12. **Создать дашборды в Grafana:**
    - **Overview** — общая картина системы: RPS, error rate, latency всех сервисов на одном экране.
    - **Per-service** — детальный дашборд для каждого сервиса (Shop API, CRM API, MES API).
    - **Databases** — состояние Shop DB и MES DB: connections, query latency, размер.
    - **RabbitMQ** — глубина очередей, DLX, consumer lag, publish/consume rate.
    - **Business metrics** — заказы по статусам, время обработки, источник (B2C/B2B).
13. **Настроить алерты:**
    - Error rate > 5% на любом API — severity: critical.
    - Latency p99 > 5s на Shop API / CRM API — severity: warning.
    - Latency p99 > 10s на MES API (дашборд) — severity: warning.
    - DLX messages > 0 — severity: critical (каждое сообщение в DLX — потенциально потерянный заказ).
    - RabbitMQ consumers count = 0 на любой очереди — severity: critical.
    - CPU > 80% на любом инстансе — severity: warning.
    - DB connections > 80% от max_connections — severity: warning.
    - Disk usage > 80% — severity: warning.

## 5. Показатели насыщенности (дополнительное задание)

Насыщенность (Saturation) — это степень использования ресурса, при которой начинается деградация. Ниже описаны пороговые значения и действия при их превышении.

### 5.1. CPU на EC2-инстансах (Shop API, CRM API, MES API)

| Порог | Значение | Обоснование |
|---|---|---|
| Warning | **70%** | Приложение начинает конкурировать с ОС за ресурсы. При всплесках нагрузки запас отсутствует. |
| Critical | **85%** | Высокая вероятность деградации latency. При 85%+ CPU увеличивается время ожидания в runqueue. |

**Действия при превышении:**
- Warning: автоматический алерт в канал дежурного, создание тикета на анализ причин.
- Critical: алерт + звонок дежурному (PagerDuty/VictorOps). Немедленное расследование. При сохранении > 15 минут — добавление инстанса (горизонтальное масштабирование).

### 5.2. Memory Utilisation на EC2-инстансах

| Порог | Значение | Обоснование |
|---|---|---|
| Warning | **75%** | Для JVM (Shop API, CRM API): GC начинает работать агрессивнее, увеличивая паузы. Для .NET (MES API): аналогичная деградация. |
| Critical | **90%** | Высокий риск OOM. ОС может начать использовать swap, что катастрофически снижает производительность. |

**Действия при превышении:**
- Warning: создание тикета, анализ heap dump, проверка на утечки памяти.
- Critical: алерт + звонок дежурному. Рестарт сервиса, если утечка подтверждена. Увеличение RAM или добавление инстанса.

### 5.3. Database connections (Shop DB, MES DB)

| Порог | Значение | Обоснование |
|---|---|---|
| Warning | **70%** от `max_connections` | Запас для пиковых нагрузок. При 70% новые запросы могут начать ожидать в очереди пула соединений. |
| Critical | **85%** от `max_connections` | При исчерпании пула — отказ в обслуживании для новых запросов. |

**Действия при превышении:**
- Warning: создание тикета, анализ idle и idle_in_transaction соединений. Для Shop DB: проверить баланс соединений между Shop API и CRM API.
- Critical: алерт + звонок дежурному. Принудительное закрытие idle-in-transaction соединений. Увеличение `max_connections` или внедрение pgBouncer (connection pooler).

### 5.4. RabbitMQ — глубина очередей

| Порог | Значение | Обоснование |
|---|---|---|
| Warning | **1000 сообщений** в очереди | Consumer-ы не успевают обрабатывать. При текущей нагрузке очередь не должна содержать более нескольких десятков сообщений. |
| Critical | **5000 сообщений** в очереди | Consumer-ы серьёзно отстают. Высокий риск потери заказов при переполнении или таймауте. |

**Действия при превышении:**
- Warning: алерт дежурному, проверка состояния consumer-ов (MES API, CRM API).
- Critical: алерт + звонок, немедленная проверка consumer-ов. Если consumer-ы работают — увеличить количество consumer-ов (масштабирование сервиса). Если consumer упал — перезапуск.

### 5.5. Disk usage (БД, RabbitMQ)

| Порог | Значение | Обоснование |
|---|---|---|
| Warning | **70%** | При текущем линейном росте (+100 заказов/месяц) диск будет заполняться предсказуемо. 70% — время для планового расширения. |
| Critical | **85%** | PostgreSQL может перестать принимать записи при заполнении диска. RabbitMQ начнёт сбрасывать сообщения. |

**Действия при превышении:**
- Warning: создание тикета на расширение диска. Анализ возможности архивации старых данных.
- Critical: алерт + звонок дежурному. Экстренное расширение диска через Yandex Cloud console. Удаление старых логов/temp-файлов как временная мера.

### 5.6. Latency p99

| Порог | Значение (p99) | Обоснование |
|---|---|---|
| Warning (Shop API) | **3 секунды** | Пользователь онлайн-магазина ожидает быструю загрузку. > 3s — потеря конверсии. |
| Warning (MES API, дашборд) | **5 секунд** | Операторы уже жалуются на медленную загрузку. |
| Critical (любой API) | **10 секунд** | Фактическая неработоспособность для пользователя. |

**Действия при превышении:**
- Warning: создание тикета, анализ медленных запросов (slow query log в БД, profiling API).
- Critical: алерт + звонок. Немедленный анализ: проверка CPU/RAM, состояния БД, глубины очередей RabbitMQ. Рассмотрение экстренного масштабирования.
